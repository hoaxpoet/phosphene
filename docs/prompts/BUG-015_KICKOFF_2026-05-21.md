# BUG-015 Kickoff — Wire `applyLiveUpdate(...)` to the runtime audio path

Hand this to a new Claude Code session verbatim. Do not summarise.

## What this is

[BUG-015](../QUALITY/KNOWN_ISSUES.md#bug-015) was filed 2026-05-20 by the [CA.4 Orchestrator capability audit](../CAPABILITY_REGISTRY/ORCHESTRATOR.md). Severity P1, domain `pipeline-wiring`. The headline:

> `VisualizerEngine.applyLiveUpdate(...)` has zero production call sites. `DefaultLiveAdapter.adapt(...)` and `DefaultReactiveOrchestrator.evaluate(...)` are fully implemented and unit-tested but never invoked at runtime — the entire Phase 4.5 / 4.6 runtime adaptation pipeline is dead in production despite both increments ✅ in `ENGINEERING_PLAN.md`.

This increment is **the fix**. Diagnosis from the audit's grep evidence has already happened (the failure mode is structural — no instrumentation needed); this session implements the wire, ships the regression test, and validates against a real-music session capture.

## Why this is next

Three reasons:

1. **Product impact.** "The AI Orchestrator has planned the entire visual session and adapts as the music unfolds" is the load-bearing product claim from `CLAUDE.md` top. The plan half works; the adaptation half does not. Until the wire lands, the product runs as a static playlist with no live response to detected boundaries or mood shifts. The QR.2 / D-080 mood-override cooldown + reactive `liveStemFeatures` work is unreachable; the L diagnostic-hold / capture-mode grace window / `wait_for_completion_event` mood-override-suppression machinery has nothing to suppress.

2. **Test/prod parity.** All 16 Orchestrator test files pass green (`LiveAdapterTests`, `ReactiveOrchestratorTests`, `DiagnosticHoldTests`, `StemAffinityScoringTests`, etc.) because they invoke the orchestrators directly with fabricated inputs. The tests do not catch BUG-015 because they bypass the App-layer entry point. This is exactly the kind of test/prod gap CLAUDE.md's "Test in the production-grade rendering pipeline. No shortcuts." rule was written for (promoted 2026-05-18 after AV.1 / AV.2 / AV.2.1 each shipped green but produced painterly smear in live playback). BUG-015 is the orchestrator-side analog of that lesson.

3. **CA.4-FU-1 + CA.1-FU-1 are natural co-travellers.** CA.4-FU-1 (demote dead `DefaultLiveAdapter.transitionPolicy` field) is < 5 LOC and bundles naturally with the BUG-015 fix if the fix touches `LiveAdapter.swift`. CA.1-FU-1 (gate the per-frame `StructuralAnalyzer` chain to prep-time only) is **fully decoupled** from BUG-015 and is the audit's recommended standalone ship — but the BUG-015 diagnoser has the freshest context for deciding whether to source `liveBoundary` from `pipeline.latestStructuralPrediction` (re-enabling the per-frame chain) or from `StructuralPrediction.none` (keeping the chain off). The choice of prediction source determines whether CA.1-FU-1 stays separately ship-able or folds into BUG-015's fix.

## Hard rules for this fix

1. **Multi-increment protocol per CLAUDE.md.** This is P1. Per "Multi-increment process for P0/P1": instrumentation → diagnosis → fix → validation → release notes. The audit's grep already serves as instrumentation; the failure mode is structural and falsifiable. **Diagnosis + Fix + Validation collapse here**, but only with the caveats:
    - Write the regression test BEFORE the fix (verification criteria #1 from the BUG-015 entry).
    - The fix lands as one commit; validation as a second commit (release notes + KNOWN_ISSUES.md Resolved field update).
    - If during the work the diagnoser finds the failure is more complex than the audit's read (e.g., the wire WAS once there and was removed; or the analysis-queue site is structurally hostile to calling `applyLiveUpdate`), stop and report — do not force-fit a wire.

2. **Test BEFORE the fix.** The regression test that catches BUG-015 must be written, must fail against the current state, then must pass after the fix. Same shape as the Aurora Veil multi-frame harness rule promoted to CLAUDE.md 2026-05-18. The shape:
    - A new integration test under `PhospheneEngine/Tests/.../Integration/` (or a new App-side test target if Integration doesn't suit) that exercises a 30 s reactive session against synthetic FeatureVectors and asserts at least one `reactiveOrchestrator.evaluate(...)` invocation happens after the 15 s listening window.
    - **Or**, equivalently: a test that asserts `applyLiveUpdate(...)` is called at least once during a synthetic playback loop driven by the existing test harness. The exact shape depends on how you choose to wire the call.
    - **The cooldown regression test is mandatory**: assert that N consecutive mood-divergence events at < 30 s intervals on the same track land ≤ 1 override. The cooldown machinery is already in place at `LiveAdapter.swift:362-381`; the test confirms the fix doesn't bypass it.

3. **Preserve the 30 s per-track mood-override cooldown.** The cooldown is enforced inside `DefaultLiveAdapter.cooldownAdaptation(...)`. The App-layer wire MUST call `applyLiveUpdate(...)` at a rate where the cooldown can do its job. Lowest acceptable cadence: 1 Hz. Highest acceptable cadence: 10 Hz (above 10 Hz, the cooldown still works but each suppression call costs an NSLock acquisition + dictionary lookup; not catastrophic, but wasteful). Aim for 1–5 Hz.

4. **Choose the prediction source explicitly.** Two options:
    - **(a) `pipeline.latestStructuralPrediction`** — real per-frame predictions from the `MIRPipeline.process` chain. **This re-enables the per-frame `StructuralAnalyzer` chain CA.1 flagged as audio-callback-CPU-wasted.** CA.1-FU-1's option (a) (gate the chain to prep-time only) becomes incompatible with this choice; CA.1-FU-1 collapses into BUG-015's resolution.
    - **(b) `StructuralPrediction.none`** — sentinel. The `confidence: 0` value means `DefaultLiveAdapter.evaluateBoundaryReschedule(...)` always short-circuits (`guard liveBoundary.confidence >= 0.5` at LiveAdapter.swift:239). Boundary rescheduling never fires; only the mood-override path runs. **This pairs cleanly with CA.1-FU-1 option (a)** — the per-frame chain stays gated off; CA.1-FU-1 ships as a separate increment.

    The choice has product implications: option (a) is the higher-leverage product behaviour (live boundary rescheduling actually fires) but costs the per-frame CPU; option (b) is the lower-CPU-cost path but leaves boundary rescheduling structurally disabled. **State the choice explicitly in the commit message and in the BUG-015 Resolved field.** Document the rationale.

5. **Validate against a real-music session capture.** After the fix lands and tests pass, run a real session (Spotify-prepared or ad-hoc, ≥ 1 minute) and confirm `~/Documents/phosphene_sessions/<ts>/session.log` contains at least one line from the `Orchestrator:`, `LiveAdapter:`, or `Reactive` log-line family. Today: zero such lines. **A passing test alone is not validation per CLAUDE.md "Test in the production-grade rendering pipeline" rule** — the session-log evidence is mandatory.

6. **Stop and report instead of forging ahead** when:
    - The fix would require touching `VisualizerEngine+Audio.swift` in ways that affect the analysis-queue's existing audio-feature path (`MIRPipeline.process`, `pipeline.setMood`, etc.). Adding the call is permitted; restructuring the analysis loop is scope creep.
    - The test reveals the live-adaptation path produces a behaviour Matt would not endorse (e.g., aggressive mid-track switches even with the cooldown active). Surface the behaviour for Matt's call before landing.
    - The BUG-015 root cause turns out to be different from "missing wire" — e.g., the wire IS there but is guarded behind a condition that's never true. Re-diagnose and report.
    - The choice between prediction sources (a) vs (b) is non-obvious from the read — surface for Matt's call.
    - You find a second pipeline-wiring gap in the same area (e.g., the `liveStemFeatures` omission noted in "Stretch scope" below turns out to be a deliberate non-shipping decision rather than a bug). Bring it to Matt before bundling.

## Failure modes to watch for

Specifically for BUG-015-shaped wiring work:

1. **The "94 Hz analysis-queue rate" rule is anticipated, not observed.** CLAUDE.md cites "94 Hz on the analysis queue" as the rate `applyLiveUpdate` will hit without rate-limiting. That number assumes the analysis queue's per-frame tick is the chosen site. **It may not be.** If the wire is a Combine sink on a slower-publishing publisher, the cadence may be much lower naturally. The 30 s per-track cooldown protects against the worst case (94 Hz with no rate limit); the actual rate depends on where the wire lands.

2. **`elapsedTrackTime` accounting is non-trivial.** `applyLiveUpdate(...)` accepts `elapsedTrackTime: TimeInterval` per `VisualizerEngine+Orchestrator.swift:168`. The right source is the playback clock (not wall-clock; not `MIRPipeline.elapsedSeconds`). Verify there's an existing `playbackElapsedSeconds` or similar on `VisualizerEngine` or in the audio pipeline. If there isn't, the cleanest fix is to add one — but that's a small architectural addition; surface to Matt before adding.

3. **The `trackIndex` argument requires a non-nil `currentTrackIndex`.** `VisualizerEngine.currentTrackIndex` is `@Published Int?` per D-091. During reactive mode (`livePlan == nil`), the index is meaningless — but `applyLiveUpdate` routes to `applyReactiveUpdate(...)` based on `orchestratorLock.withLock { livePlan }` being nil, so the index argument isn't read in the reactive path (verify by reading `applyReactiveUpdate(...)`). For session mode, the index matters; nil means the App-layer `currentTrackIndex` publisher hasn't yet resolved the live track (e.g., before first track-change callback). The fix should guard `currentTrackIndex != nil` before calling `applyLiveUpdate`.

4. **Combine subscriptions can leak.** If the wire is a Combine sink, store the `AnyCancellable` on `VisualizerEngine` so it doesn't drop on the next scheduling tick. `VisualizerEngine` already has `presetCompletionCancellable` per `VisualizerEngine+Presets.swift:507` — pattern-match.

5. **The cooldown machinery is OPT-IN — it returns nil from `cooldownAdaptation(...)` when the cooldown has lapsed.** The wire just calls `applyLiveUpdate(...)` at the chosen cadence; the LiveAdapter's internal cooldown logic suppresses redundant calls. **Do NOT add a separate cooldown at the wire site** — that's two cooldowns competing and is the kind of double-gating CLAUDE.md's "One audio primitive per visual layer" rule is the cross-product analog of.

6. **The fix may surface a separate boundary-rescheduling bug.** Once the wire is live, boundary rescheduling will fire for the first time. The 5 s deviation threshold (`boundaryRescheduleThreshold` per LiveAdapter.swift:158) was chosen pre-runtime; it may be wrong in practice. If the real-music session shows boundary rescheduling firing too often or never, surface the calibration question without changing the constant — that's a follow-up increment, not BUG-015 scope.

7. **Test/prod parity on the chosen wire site.** Whatever site you choose, the regression test must exercise the same dispatch path the live app uses. If the wire is in `VisualizerEngine+Audio.swift` and the test exercises `applyLiveUpdate(...)` directly without going through `VisualizerEngine+Audio.swift`, the test will pass but the wire may still be broken in production (e.g., the analysis-queue caller might be guarded behind a state check). The test must drive the audio path; ideally use the `SoakTestHarness` or a similar end-to-end fixture.

8. **Trivial-finding inflation.** The fix is fundamentally a single function call at the right site. Most of the work is the test, the validation, and the bundling decisions. Resist the temptation to refactor the analysis-queue's existing structure to "make the call site cleaner" — that's scope creep.
