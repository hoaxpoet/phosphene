# CLEAN Phase 5 — Kickoff: ML-weight load-time `sha256` integrity gate (CLEAN.5.5, [GAP-11]) — the last Phase-5 item

## Why this is next (and what just shipped under it)

CLEAN.5.4d added an **LFS-present** check: it fails the build if any weight `.bin` is still a Git-LFS pointer stub. That proves the weights are **present**. It does **not** prove they are **correct** — a truncated download, a partial smudge, bit-rot, a wrong-version checkpoint, or a tampered file all pass the present check and then feed garbage into the stem separator / beat tracker with **no error** (you get bad stems / wrong beats, not a crash). CLEAN.5.5 is the complementary layer: **content** integrity via a load-time `sha256` gate. Present (5.4d) + correct (5.5) = the weight supply chain is closed.

The audit's **G11** ("ML weight integrity / LFS supply chain", `CODE_AUDIT_2026-06-13.md:183`) was **LARGELY REFUTED** down to exactly two narrow questions: *does the loader validate the `sha256` that the manifest carries, and does the stem manifest have/match hashes?* Both verified **NO** below. This increment answers both.

## Scope

**In:** CLEAN.5.5 — a fail-loud, load-time `sha256` check for the two **on-disk** weight sets: **beat_this** (wire validation to the hashes that already exist) and **stem** (add hashes to the manifest, then validate). **Out:** Mood classifier (compiled-in arrays, no file); changing any model weights; per-frame / streaming re-validation (load-time once is the gate); the LFS-present check (shipped as 5.4d); deployment/toolchain (5.4).

## The work

### 5.5a — Validate beat_this `sha256` at load (the hashes already exist)
The cheap half — the manifest already has 161 digests; just check them.
- Add `sha256: String` to `BeatThisManifest.TensorEntry` (it's `Decodable`).
- In the tensor-load path (where the `Data` is already in hand), compute `SHA256.hash(data:)`, hex-encode **lowercase, no separators** (match the manifest format — verify against an existing digest, e.g. `aa3b1437…`), compare to `entry.sha256`, and **throw** a new `BeatThisWeightError.checksumMismatch(key:, expected:, got:)` on mismatch.
- **Done-when:** a one-byte edit to any `beat_this/*.bin` fails the load with a clear, named error; untouched weights load unchanged.

### 5.5b — Add `sha256` to the stem manifest + validate
- **Generate** a per-tensor `sha256` for the 172 stem `.bin` and add it to each entry in `Weights/manifest.json`. **Lean path:** a small script that hashes the **existing committed `.bin`** (they are the source of truth, already in LFS) and injects the field — do **not** re-run the full `tools/extract_umx_weights.py` umx extraction (a fresh extraction could produce subtly different bytes; we want to pin the bytes that ship). State which approach you took.
- Add `sha256` to the stem `WeightManifest` tensor struct (`StemModel+Weights.swift:13`+) and validate inside `loadTensor` (`:221`), right alongside the existing `expectedBytes` guard. New `StemModelWeightError.checksumMismatch(...)`.
- **Done-when:** every stem tensor entry has a `sha256`; a tampered/truncated stem `.bin` fails load; a test asserts the committed manifest hashes match the committed bytes.

### 5.5c — Regression tests + a completeness gate
- **Reject test:** a corrupted tensor is rejected with `checksumMismatch` — synthesize by copying a real `.bin` to a temp dir, flipping one byte, pointing a fixture manifest at it (or feed a bad-hash manifest). One per model.
- **Happy-path:** the existing weight-load tests stay green (valid weights still load).
- **Completeness gate:** every `.bin` on disk has a manifest entry **with a non-empty `sha256`** — for both models. This catches the real future failure mode: someone adds a weight file and forgets the hash, silently re-opening the hole.
- **CI-subset note (the 5.1 lesson):** the checksum logic is pure `Data` hashing → **GPU-free**, so the new suite **can** join the CI fast-gate allow-list in `.github/workflows/ci.yml` — *if* it doesn't construct an MPSGraph/Metal model (keep the checksum test separate from any model-build test). It does need the real LFS `.bin` present: CI has `lfs: true`; worktrees use `Scripts/bootstrap_fixtures.sh`. Add the GPU-free suite to the `--filter` list (and remember `--filter` matches the **type** name, not the `@Suite` string).

## Rules / pitfalls
- **Fail LOUD on mismatch** — never load-anyway-and-log (CLEAN.0 no-silent-skip). A wrong-weight model emits plausible-but-garbage stems/beats with no crash; the throw **is** the value of this increment.
- **Fold the hash into the existing `Data(contentsOf:)` load** — don't re-read the file. The I/O is already paid; marginal cost is hash compute only (~5 ms beat_this, ~70–140 ms stems), **one-time at model construction (session prep)**, not per-frame and not app launch. If even that is unwanted on the stem path, gate it behind a flag — but default ON.
- **Hash the committed bytes**, not a fresh extraction — the manifest must pin what ships.
- **Hex format must match** the manifest's existing lowercase-no-separator digests; assert this in a test so a future formatting change can't silently never-match.
- **Don't touch MoodClassifier** (compiled-in arrays — no file to validate).
- **`file_length`:** both `+Weights.swift` loaders are near the 400-line cap. Put shared hashing in a small `WeightChecksum.swift` (or the existing sibling) rather than shaving comments.
- **Don't weaken** the existing stem `expectedBytes` byte-count guard — `sha256` is *additional*, and the size check gives a cheaper/clearer error for the common truncation case.
