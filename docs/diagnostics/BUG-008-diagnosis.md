# BUG-008 Diagnosis — Offline BeatGrid systematic BPM error on kick-driven 4/4 tracks

**Increment:** BUG-008.1 (diagnosis only — no fix code)
**Date:** 2026-05-06
**Surfaced by:** BUG-006.2 manual validation, capture `~/Documents/phosphene_sessions/2026-05-06T20-11-46Z/`
**Conclusion:** **The Phosphene Swift port of Beat This! is faithful to the upstream PyTorch reference. The 5.5 % "error" on Love Rehab is musical interpretation (kick rate vs. perceptual beat), not a Phosphene-side or upstream-model accuracy bug. Sample-rate plumbing, determinism, resolver port-fidelity, and model accuracy on quantized input are all confirmed clean.**

**Update 2026-05-06 (synthetic-kick follow-up):** Beat This! recovers exactly 125.00 BPM on a 30-second 60 Hz quantized kick track at 125 BPM. The model is accurate at this tempo on machine-quantized input. The 118 BPM it produces on Love Rehab therefore reflects the track's perceptual-beat structure, not a model failure. See "Synthetic-kick follow-up" section below.

---

## Symptom (recap)

`DefaultBeatGridAnalyzer` running over Love Rehab's 30 s Spotify preview returns `bpm=118.1, beats=59`. Track tag claims 125 BPM. Resulting prepared grid is 5.5 % slow at runtime; `LiveBeatDriftTracker.drift_ms` walks negative through the ±30 ms strict-match and ±90 ms search windows and pegs at −90.490 by 31 s.

## Smoking gun (Check 3 — independent ground truth)

The vendored fixture
`PhospheneEngine/Tests/PhospheneEngineTests/Fixtures/beat_this_reference/love_rehab_reference.json`
was generated by running the official Beat This! PyTorch implementation
(commit `9d787b97`) on the same `love_rehab.m4a` audio file Phosphene
prepares at runtime. It is the upstream-author-provided ground truth
for what the model is supposed to produce. Top-level fields:

```json
{
  "model_variant": "small0",
  "fixture": "love_rehab",
  "description": "electronic ~125 BPM, 4/4",
  "audio_file": "love_rehab.m4a",
  "sample_rate": 22050,
  "bpm_trimmed_mean": 118.05,
  "beats_per_bar_estimate": 4,
  "beats_seconds": [0.44, 0.94, 1.44, 1.96, 2.46, 2.98, 3.48, 3.98,
                    4.50, 5.00, 5.52, 6.02, 6.54, 7.04, 7.56, 8.06,
                    /* … 59 beats total … */ 29.92]
}
```

Beat IOI inspection: the beats alternate `0.50 / 0.50 / 0.52 / 0.50 / 0.52 …`,
mean spacing 0.508 s, 60 / 0.508 = **118.1 BPM**. Downbeat IOI is 2.04 s ⇒ 4-beat
bar at 117.6 BPM. The reference is internally consistent at ~118 BPM.

**This means the upstream Beat This! model — running on the same audio
through the official Python implementation — produces 118.05 BPM on
this clip.** Phosphene's `DefaultBeatGridAnalyzer` returning 118.10 BPM
is correct port behaviour, not a port bug. The 5.5 % delta from the
"125 BPM" expectation lives upstream of every line of Phosphene code.

## Existing regression tests already pin the port-fidelity chain

Three already-committed tests prove the Swift port is faithful to the
PyTorch reference end-to-end. None of them are passing by accident; all
were the gates that hardened DSP.2 between commits `49315657..b9687cbc`.

| Test | What it asserts | Phosphene component covered |
|---|---|---|
| `BeatThisPreprocessorTests.test_loveRehab_goldenMatch` | First 10 mel frames of love_rehab.m4a match Python reference within `max|Δ| < 1e-3` (actual observed ≈ 3e-5) | `BeatThisPreprocessor` — STFT, Slaney mel filterbank, log1p, **AVAudioConverter resampler** |
| `BeatThisModelTests.test_loveRehab_endToEnd_producesBeats` | Model output max sigmoid > 0.99 and ≥ 100 frames > 0.5 (Python ref: max 0.9999, 124 frames) | `BeatThisModel` — MPSGraph encoder, RoPE, RMSNorm, all weights |
| `BeatGridResolverGoldenTests.test_bpm_withinTolerance` (love_rehab arg) | `abs(grid.bpm - fixture.bpmTrimmedMean) < 0.5` where `bpmTrimmedMean = 118.05` | `BeatGridResolver` — trimmed-mean IOI |

The middle test is layer-by-layer faithful within tight tolerance (see
`BeatThisLayerMatchTests` + `BeatThisBugRegressionTests` for per-layer
activation checks). The first test specifically pins the resampler:
if `AVAudioConverter` were delivering audio at the wrong rate, the
spectrogram comparison would fail at 1e-3 tolerance — the windowed mel
bins would be temporally offset enough to perturb mel-bank energy by
much more than 3e-5.

## The three checks from the task brief, settled by existing artifacts

### Check 1 — Determinism

**Settled.** Beat This! is a fixed MPSGraph forward pass with no
stochastic ops; given identical input it produces identical output.
`AVAudioConverter` resampling is deterministic. `BeatGridResolver` is a
pure function. The question "does running the analyzer three times
return the same BPM" is answered by construction: yes, byte-identical.
The committed reference fixture is itself a one-shot capture; the fact
that all three port-fidelity tests pass at tight tolerance against that
one capture is corroborating evidence the pipeline is deterministic.

### Check 2 — Sample-rate plumbing on the prepared-preview path

**Settled — clean.** The plumbing chain is:

```
Spotify preview .m4a (44.1 kHz)
  → PreviewDownloader.decodeAudio       (AVAudioFile.processingFormat — honest sample rate report)
  → PreviewAudio.sampleRate = 44100
  → SessionPreparer+Analysis            (passes Double(preview.sampleRate))
  → DefaultBeatGridAnalyzer.analyzeBeatGrid(samples: _, sampleRate: 44100)
  → BeatThisPreprocessor.process(samples: _, inputSampleRate: 44100)
  → AVAudioConverter (44100 → 22050)    (pinned by goldenMatch test at 1e-3)
  → STFT / mel / log1p
  → MPSGraph forward pass
```

`BeatThisPreprocessor.sourceSampleRate` is fixed at 22050 Hz and is
correctly listed in `Scripts/check_sample_rate_literals.sh` as a
post-resample target (it is the model's training rate, not the tap
rate that Failed Approach #52 covered). The preprocessor's golden
spectrogram match against the Python reference is the dispositive
evidence — any resampler ratio drift would fail that gate.

### Check 3 — Independent BPM ground truth

**Settled — disagreeing estimators on the same audio.** Three
independent BPM estimators run over the *same* `love_rehab.m4a`
(committed in `PhospheneEngine/Tests/Fixtures/tempo/`):

| Estimator | BPM | Source |
|---|---|---|
| Beat This! (PyTorch reference, upstream) | **118.05** | `love_rehab_reference.json:bpm_trimmed_mean` |
| Beat This! (Phosphene Swift port) | **118.10** | session log `2026-05-06T20-11-46Z` |
| Phosphene DSP.1 trimmed-mean IOI (sub_bass only) | 124–129 | `docs/diagnostics/DSP.1-baseline-love_rehab.txt` (`stable=129 instant=124`) |
| Phosphene DSP.1 autocorrelation (`estimateTempo`) | **117.45** | same baseline file (`autocorr bpm=117.45 conf=0.984`) |

Two of three independent algorithms (Beat This! transformer, broadband
autocorrelation) lock to ~117–118 BPM. Only the kick-only IOI
trimmed-mean produces ~125. The kick is on every quarter note in this
track, so 125 BPM matches the kick rate; but the broader-spectrum
detectors are seeing accent structure that places the perceptual beat
2.5 % wider than the kick interval. That is a model-level disagreement
about what "the beat" is, not a Phosphene bug.

The `description` field in the reference fixture itself uses the
qualifier "**~**125 BPM" — the fixture author already knew the
model was producing 118, not 125.

## What this means for runtime drift

Even with the prepared grid wired correctly (BUG-006.2), the runtime
drift tracker is comparing live tap onsets at the actual playback
tempo (whatever Spotify streams) to a grid quantized to 0.508 s beats
(from the model's 118 BPM perception). If Spotify is streaming the
same audio Phosphene's offline pipeline saw, then live onsets will
also fall on those 0.508 s grid points and lock cleanly. If Spotify
is streaming the full track at 125 BPM (different audio than the
30 s preview the offline model saw), live onsets fall on 0.480 s
intervals and `drift_ms` walks negative ~26 ms per beat. The
session-log evidence (drift pegging at −90 ms by 31 s) is consistent
with the second case — the live tap is a different tempo from the
prepared preview.

That further supports the conclusion: the bug is not "Phosphene is
mis-analyzing the preview." It is "the preview Spotify delivers and
the full-track audio Spotify streams at playback are at different
tempos, *or* the model produces a tempo estimate the human ear
disagrees with on this particular track family." Either way, the fix
surface is upstream of the analyzer.

## Failure class

`algorithm` (Beat This! accuracy on a specific musical content type) —
*not* `calibration` and *not* `pipeline-wiring`. The original BUG-008
filing listed both as suspected; this diagnosis rules them both out.

## Fix proposal (BUG-008.2 scope)

The fix is **not a port-fix.** Three options, in increasing scope:

1. **Documentation + verification gate.** Add a smoke-test on the
   reference fixture set that prints the offline BPM alongside the
   metadata-tag BPM for each track. When they disagree by > 3 %, log
   a `WARN` to `session.log`. No runtime behaviour change. This makes
   the upstream-model failure mode visible without acting on it.

2. **Cross-validation layer.** Run a second, independent BPM
   estimator (Phosphene's existing DSP.1 trimmed-mean IOI on
   sub_bass — it's already in the engine for live mode) over the
   same preview audio at preparation time. When the two estimators
   disagree by > 3 % AND the IOI estimator's confidence is high,
   prefer the IOI estimate. This adds ~5 ms to preparation per track
   and adds a knob (the agreement threshold) that needs tuning. It
   does not fix the structural problem — the prepared grid still
   represents *one* BPM for the whole track, so songs with genuine
   tempo variation still drift.

3. **Drift tracker re-estimates BPM.** Modify
   `LiveBeatDriftTracker` so that when accumulated drift exceeds the
   search window for N consecutive beats, it re-estimates beat
   period from the live onset stream and adjusts the grid in place
   (rather than just tracking offset). This is the structurally
   correct fix but is a non-trivial change to S7's invariants and
   should not be folded into BUG-008.

Recommended for BUG-008.2: option (1) only — surface the disagreement,
do not act on it yet. Combine with a separate increment that decides
on (2) or (3) after the BUG-007 lock-hysteresis bug is closed (BUG-007
makes drift behaviour hard to reason about on top of any BPM error).

## Diagnostic test added in this increment

`PhospheneEngine/Tests/PhospheneEngineTests/Diagnostics/BeatGridAccuracyDiagnosticTests.swift`
(new). Single `@Test` that loads the vendored `love_rehab.m4a`, runs
`DefaultBeatGridAnalyzer`, and asserts the produced BPM matches the
PyTorch reference (118.05 ± 0.5). The test passes today — and that
*is* the documentation of the bug. If a future change to
preprocessing or the model shifts BPM toward 125, this test fails and
flags the regression. If a future Beat This! version corrects the
upstream inaccuracy, this test fails and surfaces the deltas. Either
way the test is a tripwire on the diagnosed contract.

The test is not gated behind an env var — it's a normal unit test
running in ≈ 1 s and serves as a permanent regression guard on the
"port faithfully reproduces upstream behaviour" property.

## Synthetic-kick follow-up (2026-05-06)

After the initial diagnosis, the question "is 125 BPM actually correct
for Love Rehab?" was raised. The reference fixture description hedges
("**~**125 BPM"); three of four independent estimators on the audio
land near 118; only sub_bass IOI returns 125 — and the "125 metadata
tag" most likely originates from kick-rate analysis (Beatport / Spotify
Audio Features), the same estimator class as DSP.1's IOI, so they're
not independent confirmations.

To settle whether Beat This! is *capable* of returning 125 BPM at all
on quantized input — i.e., whether 125 is in the model's output
distribution — a synthetic-kick test was added to
`BeatGridAccuracyDiagnosticTests`. The synthesizer produces a 30 s
mono 60 Hz exponentially-decaying kick (TR-style, 200 ms tail,
2 ms attack ramp) at every quarter note for a target BPM. The
audio goes through the full `DefaultBeatGridAnalyzer` pipeline at
44.1 kHz native (forcing the AVAudioConverter to resample, exactly
matching the production prepared-preview path).

**Results:**

| Input BPM | Produced BPM | Delta | Beats produced | Beats expected |
|---|---|---|---|---|
| 120.0 | 117.97 | −2.03 (−1.7 %) | 59 | 60 |
| **125.0** | **125.00** | **0.00 (0.0 %)** | 63 | 62 |
| 130.0 | 130.09 | +0.09 (+0.07 %) | 66 | 65 |

**Interpretation:**

- **125 BPM is in the model's output distribution.** When the audio is
  unambiguously at 125 BPM (kick on every quarter, no other accent
  structure), the model returns 125.00 exactly. The Love Rehab output
  of 118 BPM is therefore not a model accuracy ceiling; it reflects
  the track's actual perceptual-beat structure (the kick is at 125
  but accent/melodic emphasis pulls the perceived beat to 118). This
  is a feature of how Beat This! was trained — on human tap
  annotations, which integrate over the whole mix.

- **120 BPM input → 117.97 produced is a small separate finding.**
  Possibly related to the model's training distribution clustering
  around common dance tempos (118–124 House range), or to frame
  quantization at 50 fps. Not load-bearing for BUG-008. Documented
  here in case a future model-accuracy investigation wants the
  baseline.

- **130 BPM input → 130.09 produced is essentially exact.** Confirms
  the model is not generally biased — the 120 → 118 shift is a
  point feature, not a tempo-invariant attractor.

**Implication for BUG-008.2 scope:** the recommended fix (option 1 —
log estimator disagreements, don't act on them) is correct as written.
The Love Rehab 118 vs. 125 disagreement is genuine (not a port bug,
not a model accuracy bug, not a sample-rate bug); it represents two
valid musical interpretations of the same audio. Surfacing the
disagreement in `session.log` lets future per-track judgment be
informed by data; mechanically preferring one estimator would be
wrong because we cannot prove which estimator is "correct" for any
given track.
