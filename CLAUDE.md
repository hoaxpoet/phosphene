# CLAUDE.md — Phosphene

## What This Is

Phosphene is a native macOS music visualization engine for Apple Silicon. Before the music starts, Phosphene connects to a playlist, downloads 30-second preview clips for every track, and runs full ML-powered stem separation and MIR analysis on each. By the time the user presses play, the AI Orchestrator has planned the entire visual session — which visualizer for each track, where transitions land, and what the emotional arc looks like across the playlist. During playback, real-time audio analysis via Core Audio taps (`AudioHardwareCreateProcessTap`) refines the pre-analyzed data, and the Orchestrator adapts its plan as the music unfolds.

Phosphene does not control playback — the user starts the music in their streaming app when Phosphene signals it is ready.

See `docs/PRODUCT_SPEC.md` for the full product definition, `docs/ARCHITECTURE.md` for system design, `docs/DECISIONS.md` for rationale behind key choices, `docs/RUNBOOK.md` for build/test/CI/troubleshooting, `docs/MILKDROP_ARCHITECTURE.md` for the research findings that drive the Phase MV (Musicality) work, `docs/UX_SPEC.md` for the user-facing product UX contract (state-to-view mapping, error taxonomy, onboarding), and `docs/SHADER_CRAFT.md` for the preset authoring handbook (detail cascade, material cookbook, per-preset uplift playbook) in `docs/ENGINEERING_PLAN.md`.

## Increment Completion Protocol

Every increment — engine, preset, UX, docs, infrastructure — ends the same way. The protocol below is the standing rule. Skipping a step turns a finished increment into one that "looked finished in chat" and rots within a session or two.

**Closeout report.** At the end of every increment, produce a short report covering:

1. **Files changed** — concrete paths, grouped new vs. edited.
2. **Tests run** — which suites, pass/fail counts, any pre-existing flakes called out.
3. **Visual harness output** — when the increment is preset-facing or otherwise visually observable, include the `RENDER_VISUAL=1` per-stage / contact-sheet output paths or attach key frames. State explicitly when a change is not visually verifiable.
4. **Documentation updates** — list every doc file touched.
5. **Capability registry updates** — see below; cite the rows changed.
6. **Engineering plan updates** — see below; cite the increment ID.
7. **Known risks and follow-ups** — bounded list of what could break, what was deferred, and what the next recommended increment is.
8. **Git status** — branch, commit hash(es), `git status` clean / dirty, files staged outside the increment's scope.

**Durable learnings stay in docs.** Anything a future session will need to know — a non-obvious tuning constant, a Failed Approach, a renderer convention, a preset-author rule — goes into `CLAUDE.md`, `docs/DECISIONS.md`, `docs/SHADER_CRAFT.md`, the relevant docs/ subtree, or memory. **Do not leave durable learnings only in chat.** If the only record of "we learned X" is a paragraph in this conversation, future Claude will not have it.

**`docs/ENGINEERING_PLAN.md` is mandatory to update.** Update it whenever an increment is **completed, split, renamed, deferred, or discovered to require prerequisite work**. Each increment ID should map to a row that says what done-when looks like and whether it's done. Plans drift fast — if the plan and the code disagree, treat that as a bug in the plan.

**`docs/ENGINE/RENDER_CAPABILITY_REGISTRY.md` is mandatory to update** whenever **renderer, visual harness, certification pipeline, shader infrastructure, or preset architecture capabilities change**. New capability → new row. Capability promoted Missing → Partial → Supported → flip the status and cite the files. New blocker discovered → add it. Preset implications section gets the same treatment: what's now buildable, what's still blocked. The registry is the load-bearing document for "can preset family X be built today" — let it drift and audit work doubles in cost.

**Commit locally to `main` after tests and docs are complete.** Use the standard commit message format (`[<increment-id>] <component>: <description>`). Prefer multiple small commits within an increment over one large commit — it makes `git bisect` useful.

**Do not push to the remote without Matt's explicit approval.** Local `main` commits stay local. `git push` requires "yes, push" in the chat. This applies even when the work is clearly green and clearly Matt's request — pushing remains a separate decision.

**Pruning pass — every tenth increment.** Every tenth increment (or every two weeks, whichever fires first), run a pruning pass against the doc set. The pass is its own increment with its own closeout report and does not require per-entry sign-off — borderline calls go through the standard "stop and report" rule below.

1. **Failed Approaches:** for each entry, ask "would this rule prevent a bug today?" If no, move to `docs/HISTORICAL_DEAD_ENDS.md`.
2. **Decisions:** for each entry whose increment has shipped and is no longer cited by another active decision, move to `docs/DECISIONS_HISTORY.md`.
3. **CLAUDE.md sections:** for each section, ask "did the last 10 increments need this?" If no, consider moving to a handbook (`docs/ARCHITECTURE.md`, `docs/SHADER_CRAFT.md`, `docs/UX_SPEC.md`, or `docs/RUNBOOK.md`).
4. **Current Status section:** trim to the last 10 increments; older entries are in `ENGINEERING_PLAN.md` and `git log` already.

Pruning has a counterweight role: the "Durable learnings stay in docs" rule above is the accumulation side; the pruning pass is the retirement side. Skipping the pruning side reproduces the doc-mass problem the 2026-05-13 refactor was set up to fix (see `docs/diagnostics/DOC-REFACTOR-PLAN-2026-05-13.md`).

**Stop and report instead of forging ahead** when:

- tests fail (engine or app), even pre-existing flakes that the increment touched;
- preset acceptance gates or rubric checks fail;
- implementing the increment as written would require broader architectural changes than were authorized — pause, surface the scope, get approval before expanding;
- documentation conflicts with code that just changed (e.g. CLAUDE.md describes an API that no longer exists) and resolving the conflict requires judgement calls outside the increment's scope;
- the commit would include unrelated files (`git status` shows changes outside the increment's stated scope) — back those changes out or surface them for approval before committing.

When in doubt, write a short status update and ask. The cost of pausing to confirm is low. The cost of an increment that silently expanded scope, partially landed, or quietly skipped a doc update is high.

## Defect Handling Protocol

See `docs/QUALITY/DEFECT_TAXONOMY.md` for full severity definitions, domain tags, and failure classes. See `docs/QUALITY/BUG_REPORT_TEMPLATE.md` for the required report structure. See `docs/QUALITY/KNOWN_ISSUES.md` for the active issue tracker.

**Evidence before implementation (P0/P1/P2).** Do not modify code in response to a reported P0, P1, or P2 defect until the following are documented:

1. **Expected behavior** — observable output described in concrete terms (values, units, state names). Not implementation internals.
2. **Actual behavior** — what actually happens, with observed values. Include frequency if intermittent.
3. **Reproduction steps** — minimum reproducer. A specific track or fixture if the defect is domain-specific.
4. **Session artifacts** — relevant `features.csv` columns, `session.log` lines, contact sheet screenshot, or diagnostic dump. For beat-sync and stem-routing defects, `features.csv` beat columns and a `BeatSyncSnapshot` are the primary artifacts. For render defects, a `RENDER_VISUAL=1` contact sheet is required where available.
5. **Suspected failure class** — one class from the taxonomy (`algorithm`, `concurrency`, `api-contract`, `calibration`, `pipeline-wiring`, `resource-management`, `sample-rate`, `precision`, `test-isolation`, `sdf-geometry`, `render-state`, `regression`, `documentation-drift`).
6. **Verification criteria** — written before the fix, not after. At minimum: one automated gate + one manual check for any defect affecting musical feel or visual fidelity.

**Fix increment obligations.** Every fix increment must:
- Update `docs/QUALITY/KNOWN_ISSUES.md` — fill in the `Resolved` field and commit hash.
- Update `docs/RELEASE_NOTES_DEV.md` — add or extend the current release entry.
- Not skip these updates under "it's obvious from the commit."

**Multi-increment process for P0/P1.** Unless a defect is trivial (< 5 lines of change, root cause obvious from existing artifacts, no architectural risk — requires Matt's explicit approval to collapse), the fix process uses separate increments:

1. **Instrumentation** — add logging, diagnostic capture, or test infrastructure to expose the failure. Commit and stop.
2. **Diagnosis** — reproduce from artifacts, identify root cause, document in `KNOWN_ISSUES.md`. Do not write fix code in this increment.
3. **Fix** — implement the fix, add or extend regression tests.
4. **Validation** — run full test suite, produce required domain artifacts, perform manual validation where mandated.
5. **Release notes** — update `RELEASE_NOTES_DEV.md`, mark resolved in `KNOWN_ISSUES.md`.

Trivial P1 defects may collapse steps 1–4 into one increment. State this explicitly in the commit message and in `KNOWN_ISSUES.md`.

**Domain-specific artifact requirements.** These domains require diagnostic artifacts before and after fix work:

- **Beat sync / tempo** (`dsp.beat`): `features.csv` beat-sync columns (`lock_state`, `grid_bpm`, `drift_ms`, `barPhase01_permille`), SpectralCartograph mode label capture, and `BeatSyncSnapshot` data from a real music session. Minimum: Love Rehab at 125 BPM.
- **Stem routing** (`dsp.stem`): `stems.csv` showing non-constant deviation-field values across 500+ frames, plus manual observation that visual response feels musically connected.
- **Preset fidelity** (`preset.fidelity`): contact sheet from `RENDER_VISUAL=1` compared against `docs/VISUAL_REFERENCES/<preset>/` reference images. Anti-references must be explicitly checked (see Failed Approach #48).
- **Render pipeline** (`renderer`): `PresetRegressionTests` golden hash before and after; Metal GPU trace if frame budget is affected.

**Manual validation is required for:**
- Musical feel: beat alignment, stem-visual coupling, tempo tracking. Automated tests prove pipeline correctness; they do not prove the result feels musical. These judgments require listening at normal volume.
- Visual fidelity: M7 review for any preset approaching certification. Matt's approval is required; no automated metric substitutes.
- UX flow: any change to the session lifecycle or playback chrome. Walk the affected flow end-to-end; do not rely solely on unit tests of view models.

## Audio Data Hierarchy — The Most Important Design Rule

**Learned the hard way in the Electron prototype. Beat-dominant designs feel out of sync. Continuous-energy-dominant designs feel locked to the music. Non-negotiable.**

### Layer 1: Continuous Energy Bands (PRIMARY VISUAL DRIVER)
`bass`, `mid`, `treble` (3-band) and 6-band equivalents. Zero detection delay. Feedback zoom, rotation, color shifts, geometry deformation — all driven primarily by these.

### Layer 2: Spectrum and Waveform Textures (RICHEST DATA)
512 FFT magnitude bins + 1024 waveform samples → GPU as buffer data, not scalars. Modern GPUs process 512+ bins per fragment.

### Layer 3: Spectral Features (DERIVED CHARACTERISTICS)
Centroid, flux, rolloff, MFCCs, chroma. Modulate color temperature, complexity, scene behavior.

### Layer 4: Beat Onset Pulses (ACCENT ONLY — NEVER PRIMARY)
±80ms jitter from threshold-crossing. Feedback amplifies jitter. Must NEVER be the dominant motion driver.

### Layer 5a: Pre-Analyzed Stems (AVAILABLE FROM FIRST FRAME)
From 30-second preview clips. Available instantly on track change via StemCache. Not time-aligned with live playback.

### Layer 5b: Real-Time Stems (REPLACES 5a AFTER ~10 SECONDS)
From live Core Audio tap via MPSGraph. Time-aligned with playback. Crossfades with 5a.

**Rule of thumb:** `base_zoom` and `base_rot` (continuous energy) should be 2–4× larger than `beat_zoom` and `beat_rot` (onset pulses).

### Cold-Start Phase Contract

Production is the pre-impl baseline. The section below has been rewritten to describe that state. See `docs/RELEASE_NOTES_DEV.md` `[dev-2026-05-26-b]` for the revert narrative.

What the system can and cannot claim about beat synchronicity at the start of a new track. The original Phase CS bar (±50 ms / 90 % from frame 1) is empirically falsified across six iterations (CS.1 → CS.1.y.1 → CS.1.y.2 → CS.1.y.2-redo r1 → r2 → BSAudit.3.impl, 2026-05-22 → 2026-05-25) and is retired per Matt's Choice A decision 2026-05-25. The BSAudit.3.impl runtime that Choice A's "doc-only closeout" preserved was itself reverted the same evening; production is the pre-impl baseline. See Failed Approach #69 below.

**The premise that failed.** "There is some automated signal in the first ~3 s of tap audio that reliably tells us the audible beat phase of a novel track." Six iterations exhausted the available signals (sub-bass onsets, Beat This! on short windows, Beat This!@15s, broadband flux peaks). None converged on > 70 % of the catalog. The premise is retired.

**What production delivers at cold-start (post-2026-05-25 revert).**

- **Continuous-energy modulation from frame 1.** `bass`, `mid`, `treble`, and the deviation primitives (`bassRel`, `bassDev`, etc.) are active immediately. Presets driven by Layer 1 continuous energy show full response from the first frame. (Unchanged by the impl + revert — Layer 1 is upstream of BSAudit.3.impl's scope.)
- **Cached `BeatGrid` install at track start.** Beat This! on the 30 s Spotify preview → BPM + meter are reliable. The grid is installed via `MIRPipeline.setBeatGrid(_:initialDriftMs:)` (the pre-impl API; `installBPMPrior(bpm:character:beatsPerBar:)` was removed by the revert). The grid's phase is correct on the preview clip but may be wrong-phase on the user's live audio — Beat This! is cross-capture-unstable on 5–6 of 10 audit-catalog tracks (BSAudit.2 finding).
- **`LiveBeatDriftTracker` (pre-impl form).** Operates on sub-bass onset events with an EMA over per-event residuals. No BPM-prior phase acquisition, no broadband-peak detector, no confidence accumulator. The BSAudit.3.impl.2 rewrite was reverted on 2026-05-25.
- **`GridOnsetCalibrator`.** Pre-track sub-bass onset → grid alignment helper. Reinstated by the 2026-05-25 revert of BSAudit.3.impl.3 (which had retired it).
- **Ungated beat accents.** `beatBass / beatMid / beatTreble / beatComposite` flow at full amplitude from frame 1 based on the cached grid. There is no `accentConfidence` field on `FeatureVector`; the BSAudit.3.impl gating layer is not present.

**What that means in practice.**

- On tracks where the cached BeatGrid is on-phase with live audio (the majority of catalog), beat-driven presets fire reasonably from frame 1. Steady-state lock improves as the `LiveBeatDriftTracker` EMA converges on per-onset residuals.
- On tracks where the cached grid is wrong-phase (cross-capture-unstable tracks per BSAudit.2 — 5–6 of 10 in the audit catalog), beat accents fire wrong-phase from frame 1 with no gating to suppress them. This is the "accent over-firing during cold-start" state that BSAudit.3.impl was designed to address; the trade-off accepted at the 2026-05-25 revert is simpler runtime + restored `GridOnsetCalibrator` over the impl's gating mechanism.

**What production does NOT deliver.**

- **±60 ms perceptual sync within 3 s on a novel track.** The original product bar of `docs/BPM_ANCHORED_PHASE_ACQUISITION_DESIGN_2026-05-24.md` §1, attempted across six iterations including BSAudit.3.impl, is structurally unachievable from live tap audio alone. Two empirically-characterized failure modes (`docs/diagnostics/BSAUDIT_3_VALIDATE_3_DIAG_2026-05-25.md`):
  - **Wrong-anchor lock.** First broadband peak fires on pre-beat content (pad swells, vocal entries, hi-hat lead-ins) on 5 of 10 catalog tracks; confidence still climbs to ≥ 0.9 because periodic broadband content at quarter-note rates reinforces *any* phase that matches the period prior, not just the on-beat one. Design §9.1's mitigation ("confidence will not climb on a wrong anchor") empirically does not hold. (This failure mode characterized the *impl*; the structural limit it exposes applies to any short-window signal source.)
  - **Slow-anchor lock.** On tracks with sparse early-window onsets, the impl's confidence accumulator plateaued below threshold; the few accents that fired had wide residual distributions. In the post-revert pre-impl baseline, accents fire on the cached grid regardless — the failure mode is "wrong-phase firing" rather than "no firing."
- **Cross-capture-stable verification.** Beat This!-on-tap is per-capture-stable but cross-capture-unstable on 5-6 of 10 catalog tracks (BSAudit.2 finding). Any verifier metric anchored to Beat This!-on-tap is informative within one capture but not as an absolute judge of cross-build fix claims. The `ColdStartVerifier --accent-window-pass-rate` mode is useful for relative comparison; it is not a perceptual-sync close gate on its own.
- **Confidence-gated accent suppression.** The BSAudit.3.impl soft-ramp warmup (`accentConfidence` multiplier on `beatBass/Mid/Treble/Composite`) was reverted. Presets needing cold-start accent suppression must implement it themselves (e.g., time-since-track-start envelope, deviation-primitive gating per D-026).

**What this means for preset authoring.**

- **Drive primary motion from Layer 1.** Continuous energy is the bedrock and is reliable from frame 1. Unchanged from the existing Audio Data Hierarchy rule.
- **Beat-pulse fields are accents, not primary motion.** Unchanged. Membrane's per-kick shockwave, Lumen Mosaic's cell dance, etc. are accent layers. The pre-impl baseline fires these from the cached grid from frame 1, ungated — wrong-phase tracks will fire wrong-phase accents.
- **Accent gating is NOT automatic.** (CHANGED post-revert.) Presets that consume `beatBass / beatMid / beatTreble / beatComposite` see full-amplitude fields from frame 1; there is no soft-ramp warmup at the `FeatureVector` layer. Presets that need cold-start accent suppression must implement it themselves (per-preset time-since-track-start envelope, deviation-primitive gating per D-026, or similar).
- **`beatPhase01` / `barPhase01` are NOT gated** (no change). Continuous-phase fields can drive non-pulse motion at any confidence level. The acquired phase may be off-beat in the cold-start window; presets relying on `beatPhase01` for tight per-beat timing should already be using it for continuous motion (color cycles, vocal-pitch tracking) where small phase errors read as small motion offsets rather than wrong-beat firings.
- **Do not iterate further on automated cold-start beat-phase derivation.** See Failed Approach #69. Any future work in this space requires a new design premise (human-tap reference per BSAudit-FU-5 Path B, full-track local-file analysis, manual per-track calibration UX) — not another short-window signal.

**Historical: the BSAudit.3.impl attempt.**

The BSAudit.3.impl architecture (commits `efaf8cb4..30d032ea`, shipped 2026-05-24, reverted 2026-05-25 evening) added BPM-prior phase acquisition, a broadband-peak detector, `RhythmCharacter` metadata, and confidence-gated accents (`accentConfidence ∈ [0,1]` multiplying `beatBass / beatMid / beatTreble / beatComposite` in `MIRPipeline.buildFeatureVector`). Validate.3 showed 4/10 PASS-firing|degraded on the new metric; diag.1 root-caused two failure modes (wrong-anchor lock, slow-anchor lock). ## Code Style

- Swift 6.0, `SWIFT_STRICT_CONCURRENCY = complete`. `async`/`await` and actors. Avoid raw `DispatchQueue` except for Accelerate/vDSP.
- Shared types: `Sendable`. Audio frame types: `@frozen`, SIMD-aligned.
- `NSLock.withLock {}` from synchronous contexts only. For types mixing sync callbacks with async API, use `@unchecked Sendable` class with NSLock.
- No `print()`. Use `os.Logger` via `Shared/Logging.swift`.
- SwiftLint: `force_cast`/`force_try`/`force_unwrapping` → error. `file_length` warning at 400. `cyclomatic_complexity` warning at 10.
- All `public` API has `///` doc comments. Every file uses `// MARK: -` dividers.
- Protocol-first design. Every injectable dependency has a protocol. Tests use doubles from `TestDoubles/`.
- **URLProtocol stub tests require `@Suite(.serialized)`** (U.10 learning). Swift Testing runs suites in parallel by default. A suite that uses a global `nonisolated(unsafe) static var handler` on a `URLProtocol` subclass must be annotated `@Suite(.serialized)` — otherwise one test's handler bleeds into another test's in-flight URL session on a background thread. Discovered when 5 of 9 `SpotifyTokenProviderTests` returned `HTTP 400` instead of 200 during parallel execution.
- **When adding enum cases to a VM's state type, update every `switch` in the corresponding view simultaneously.** New cases that are `@ViewBuilder`-switch arms in a different file will silently fail to compile only when the app target builds (not the engine SPM target). Discovered when `.privatePlaylist` and `.authFailure` were added to `SpotifyConnectionState` but `SpotifyConnectionView`'s exhaustive switch was not updated — the engine test suite passed but the app build failed.
- **App-layer services use `Logger(subsystem:category:)` directly, not `Logging.session`.** The engine `Shared/Logging.swift` module's `Logging.session` logger is only available within `PhospheneEngine`. App-layer files in `PhospheneApp/` must `import os.log` and instantiate their own `Logger(subsystem: "com.phosphene.app", category: "...")`. Discovered during U.11 when `SpotifyKeychainStore` and `SpotifyOAuthTokenProvider` referenced `Logging.session` and failed to build. (U.11)
- **New app-layer source files must be registered in Xcode project.pbxproj across all four sections.** Files on disk that are not in the project file cause `cannot find type` build errors. Four sections must all be updated: `PBXBuildFile` (build file entry), `PBXFileReference` (file reference), `PBXGroup` (parent group membership), and `PBXSourcesBuildPhase` (target sources list). The project uses alphabetical UUID prefixes (N10xxx / N20xxx were the next-available block in U.11). Verify after adding files: `xcodebuild -scheme PhospheneApp build` will fail immediately if any section is missing. (U.11)

If the user's Audio MIDI Setup runs at 96 kHz, beat windows and BPM math are off by ~2.18×. Set Audio MIDI Setup to 48 kHz to match (RUNBOOK).

31. **Absolute thresholds on AGC-normalized energy** (e.g. `smoothstep(0.22, 0.32, f.bass)`): AGC's denominator (running-average) moves with mix density, so the same acoustic kick reads different values across tracks or across sections of one track. Six VL iterations (v3→v4.2) hit this repeatedly. Drive from deviation instead — `f.bassRel`, `f.bassDev` (D-026, MV-1). See `docs/MILKDROP_ARCHITECTURE.md` for the full diagnosis.
32. **Driving ray-march preset visuals from instantaneous audio alone**: feedback is the mechanism that turns simple audio into compound musical motion (Milkdrop's core insight). Ray-march presets rendered from scratch each frame can only show instantaneous audio state — no accumulation, no "breathing", no musicality regardless of how clever the shader drivers are. Fixed in MV-2: the `mv_warp` render pass (D-027) provides per-vertex feedback accumulation. See VolumetricLithograph for the reference implementation.
33. **Free-running `sin(time)` oscillation in organic preset motion**: A `sin(time * k)` term runs at a fixed cycle rate regardless of music, making the visual feel mechanical and disconnected from the audio. Observed in Arachne v1 (session 2026-04-21T13-26-38Z): hub throb `sin(time*9)` ≈ 1.4 Hz oscillation ignored tempo entirely; strand quiver `sin(dist*15 - time*4.8)` scrolled continuously even at silence. Fix: replace with beat_phase01-locked phase (`sin(dist*k - beat_phase01*2π)` = exactly one wave per beat) or beat-anticipation amplitude (`smoothstep(0.75, 1.0, beat_phase01)`). If BeatPredictor has no stable estimate, gate on `bassDev > 0` so motion only fires on above-average transients. **Scope:** applies to motion the viewer expects to land with the music — primary motion in active visual subjects (web hubs, particle pulses, hero geometry). Does **NOT** apply to ambient/atmospheric motion: dust mote drift, slow mood gradients, `accumulated_audio_time`-driven palette cycling, sky-band low-frequency `fbm4` modulation. These are intentionally untethered to instantaneous audio and read as scene texture, not music response.
34. **`abs(fract(x) − 0.5)` as an SDF fold for periodic structures (spiral threads, ring bands)**: This formula gives 0 at integer positions (in the GAPS between threads) and 0.5 at half-integer positions (ON the threads). It is the inverse of a correct distance field — coverage computed from it is maximal everywhere threads are NOT, filling the entire area instead of drawing thin strands. Use `min(fract(x), 1 − fract(x))` which correctly gives 0 ON the thread and 0.5 in the gaps. Bit both Arachne and Gossamer during Increment 3.5.10/3.5.11 and caused both to render as filled discs. The visual tells: perfectly uniform lit region where web should be, no strand structure visible.

35. **Single-octave noise for hero surfaces**: 1–3 octaves of Perlin or fBM reads as primitive. Real surfaces have variation across many spatial frequencies simultaneously. Minimum 4 octaves for any hero surface; 8 octaves (`fbm8` utility) for terrain or cloud fields. Observed across every preset iteration before Phase V. See `SHADER_CRAFT.md §3`.

36. **Uniform-albedo-per-material presets**: A constant `float3 albedo` anywhere on a hero surface reads as clipart. Real surfaces have per-point variation. Drive albedo through `fbm8` or `worley_fbm` at minimum, or through a cookbook material recipe from `SHADER_CRAFT.md §4` that already layers variation.

37. **Constant roughness**: `roughness = 0.3` across a surface reads as CGI-plastic. Vary roughness spatially via noise. Even 10% variation breaks the plastic look. Silk, metal, wet stone, ferrofluid — all need spatially varied roughness.

38. **Grey fog**: Fog color matching scene palette (sky, horizon, mood) reads as atmosphere. Grey fog reads as a printing defect. Always tint fog to match scene, not to a neutral middle-gray.

50. **Fusing onset events across bass bands for IOI-based tempo**: `recordOnsetTimestamps` originally consumed `bandFlux[0] + bandFlux[1]` (sub_bass + low_bass summed under a single P75 threshold). A 60 Hz kick fires flux events in *both* bands at slightly different frames — the kick fundamental peaks first, the harmonic peaks one or two FFT-hop frames later. Per-band cooldowns in `detectOnsets` are independent (400 ms each), so the OR-stream produces alternating 18-frame (418 ms) and 19-frame (441 ms) IOIs for a true 441 ms beat. Combined with histogram-mode bucketing (Failed Approach #51), this systematically biased BPM estimation toward the faster bucket. Fix in DSP.1 (D-075): single-band sourcing only — `guard onsets[0] else { return }`. Tracks with empty sub_bass fall through to the autocorrelation tempo path. Validated on so_what (152 → 136 BPM, true 136) and love_rehab (117 → 124 BPM, true 125).

51. **Histogram-mode picking of integer-rounded BPM buckets**: Bucketing IOIs by `Int(round(60/ioi))` — integer BPM — gives non-uniform bucket widths in period space. The 144 BPM bucket spans periods [414.4, 419.6 ms] (5.2 ms wide); the 136 BPM bucket spans [437.5, 442.7 ms] (5.2 ms wide); but the 80 BPM bucket spans [740.7, 759.5 ms] (18.8 ms wide). When FFT-hop-quantized IOIs cluster at adjacent frame counts, the higher-BPM buckets accumulate more events even when the underlying tempo would map cleanly to a lower-BPM bucket — picking the histogram mode systematically biases toward faster BPMs. Fix in DSP.1 (D-075): drop histogram-mode picking entirely, use trimmed-mean IOI in `computeRobustBPM` (median IOI → reject outliers outside [0.5×, 2×] median → mean of inliers → `60/meanIOI`). The histogram is still built for the diagnostic dump, but never consumed by the BPM picker. Future tempo work should not re-introduce histogram-mode picking.

52. **Hardcoded `44100` sample rate constants in any code path that should consume the live tap rate** (recurrence of Failed Approach #29 at the *code* layer rather than the *Audio MIDI Setup* layer). 53. **Summing AGC-normalized stem energies for orchestrator scoring**: `DefaultPresetScorer.stemAffinitySubScore` summed `stemEnergy[stem]` across declared affinity stems and clamped to [0, 1]. AGC centers each stem at 0.5, so any preset declaring 2+ matching affinities saturates at ~1.0 on most music — two presets with totally different declared affinities score nearly identically. The 25% score-weight slot for stem affinity did no work; preset selection was carried entirely by the mood (0.30) and section (0.25) weights. Same root cause as Failed Approach #31 in the orchestrator instead of the shader. Fix in QR.2 (D-080): score against deviation primitives (`stemEnergyDev[stem]` from D-026 / MV-1, already on `StemFeatures` floats 17–24) and use the *mean* of `max(0, stem_dev)` over declared affinities, not the clamped sum. Two presets with disjoint affinities now produce score gaps ≥ 0.3 on tracks where one matches and the other does not.

Combined with the pre-QR.2 `stemAffinitySubScore` (Failed Approach #53), presets with declared affinities scored 0 on every reactive track while presets without scored 0.5 — exact inversion of intent. 61. **Implementing "moving colored beams reflected on a near-mirror surface" as discrete point lights with physical inverse-square falloff** (V.9 Session 4 Ferrofluid Ocean §5.8 stage rig, shipped 2026-05-13 + M7-failed same day, rescue authored as V.9 Session 4.5): the §5.8 spec described "moving colored beams reflected on the ferrofluid surface" and I implemented it as 4 point lights orbiting at altitude 6 / radius 4 with Cook-Torrance per-light dispatch. The §5.8 musical contract (vocals_pitch → palette phase, drums_energy_dev → intensity envelope, arousal → orbit speed) is preserved across the paradigm change; only the GPU consumption changes from "Cook-Torrance per-light loop" to "procedural sky sample at reflection vector."

Each of the three layers had nominal audio routing (droplets ← `bass_att_rel`, meso ← `mid_att_rel`, micro-normal = always-on substrate identity) but none produced a visual moment a listener could point at and pair with a musical moment. The micro-normal in particular destroyed the substrate's mirror identity (scattering thin-film specular) — the layer's claimed "tactile identity" role contradicted the references' "razor-sharp specular on near-mirror" character. The droplets fired across the entire spike field regardless of apex height (the prompt spec said "spikeHeight > 0.6 × max" but I omitted the gate), producing visible stippling everywhere. **The Authoring Discipline rule "articulate the musical role before authoring anything" applies at LAYER scope too, not just at PRESET scope.** Before adding any decoration layer, the one-sentence check is: *"can a listener point at a moment in the music and identify the visual response THIS layer produces?"* If the answer is "yes — this fbm warp makes the surface choppier when mids thicken" but the rendered visual reading is "random stippling competing with the hero spike formations," the layer has failed the check. The fix is to delete the layer, not to tune its coefficients.

63. **Authoring a preset session without reading `docs/VISUAL_REFERENCES/<preset>/README.md`** (V.9 Session 4 Ferrofluid Ocean, 2026-05-13): the README for Ferrofluid Ocean carries a stylization caveat (*"None of them is a faithful rendering of 'Ferrofluid Ocean as it should appear'. Read each image only for the trait its annotation calls out"*), per-image annotations that distinguish trustable traits from disregard-this-property warnings, a mandatory traits checklist, and an explicit anti-references list. Session 4 was authored against the Session-4 prompt text alone, without consulting the README. The result shipped three decoration layers (Failed Approach #62) and a §5.8 paradigm misread (Failed Approach #61) that the README would have caught — the README's `08_*` annotation explicitly warns *"the preset's beams are continuous diffuse gradients, not point sources with pillar reflections"* and the README's mandatory-traits checklist explicitly says *"chromatic content present only in the reflections (not in the underlying surface tone)"*. The session's mid-session sanity-check claim "no structural divergence from the references was observed" was wrong because the comparison was against self-judgment, not against the named reference images. **Rule:** Do not author a preset session without (a) reading the README cover-to-cover, (b) reading each reference image's annotation, (c) citing specific reference image traits in design comments where they motivate a design choice. The §V.5 Failed-Approach-#39 rule "do not author without references" extends here to "do not author without ALSO reading the README + annotations." Mid-session sanity checks must be side-by-side comparisons against the named reference images, not self-judgments of "looks reasonable."

64. **Iterating on shader fixes by first-principles guessing on a problem with known published solutions** (V.9 Session 4.5 Phase A Ferrofluid Ocean, 2026-05-13 → 2026-05-14): a per-cell "dot pattern" appeared in rendered output after the Session 4.5 lighting rebuild. I made six successive structural claims about the cause — bell-curve flat top reflects zenith → linear cone fixes it → no wait, linear cone normal-flips at cell edges → quadratic cone fixes both → still dots, must be a resolution artifact → bump test resolution → still dots → must be aurora coverage mismatch → sweep aurora band thickness 0.30 / 0.50 / 0.70 / 1.00 → still dots — each fix landed mechanically, none converged. Matt eventually pointed out (i) several of my "production won't show this" assertions were unverified, and (ii) ferrofluid rendering is a solved problem in the field and I should be doing desk research, not guessing from first principles. Desk research surfaced Robert Leitl's audio-reactive WebGL ferrofluid project (closest published reference to Phosphene's use case): the canonical technique is **smooth Voronoi** (Inigo Quilez's exponential-blend soft-min) for the height field — the C¹-continuous distance function eliminates cell-boundary normal-flip artifacts that were producing the dot pattern. One library function from a 2014 article fixed in 30 minutes what six iterations of first-principles cone-shape tuning failed to fix. **Rule:** when iterative first-principles fixes aren't converging on a problem that has known prior art in the field, *stop guessing and do desk research*. Failed Approach #49 is the structural-gap escalation; this is its discipline-rule counterpart at the implementation-technique level. Symptoms that should trigger desk research: (a) ≥ 2 successive structural fixes failed to converge; (b) the failure mode has a recognizable name in graphics literature ("dot pattern," "shading artifact at cell boundaries," "ferrofluid rendering"); (c) other implementations of the same effect exist and are findable in 1–2 web searches. The cost of desk research is small. The cost of three more iterations of guess-test-fail is large.

65. **Negotiating away components of a working reference implementation under unverified "redundancy" arguments** (V.9 Session 4.5 Phase A Ferrofluid Ocean, 2026-05-14): after desk research surfaced Robert Leitl's four-layer lighting model (Phong specular + env map + fresnel rim + iridescence palette) as the technique producing his demo's "incredible" character, I proceeded to argue each component into the trash — iridescence is "redundant with aurora," fresnel rim is "redundant with thin-film F0," Phong specular contradicts "mirror-reflects-sky." Each redundancy claim was unverified hand-wave reasoning; Matt called it out: *"I had you review Leitl's demo to get ideas and now it appears that you have rejected all of the ideas you found within it."* The components were mathematically *different operations* on different geometric domains — Leitl's demo character comes from all four together, not from any single one. **Rule:** when adopting a working reference implementation, the default is to adopt the components that produce its visual character verbatim and adapt only what differs in *context* (scale, audio routing, scene type). "Redundancy" claims against components of a working reference require rendering proof of the claim, not first-principles math. The failure mode is: anchor on prior paradigm → find reasons to subtract from reference → end up with neither paradigm working. **Tell-tale phrasing of the failure:** "I think X is redundant with Y" without having tested removal. If you catch yourself writing that sentence, you are about to repeat this failure.

66. **Test/prod parity gap: fixture and live render paths take different G-buffer branches for the same preset** (Ferrofluid Ocean rounds 50–57, 2026-05-16 → 2026-05-17). Phosphene's `RayMarchPipeline.render` chooses between `runMeshGBufferPass` and `runGBufferPass` based on whether a mesh G-buffer encoder was registered via `pipeline.setMeshGBufferEncoder(...)`. Ferrofluid Ocean's `VisualizerEngine+Presets.swift` registered a mesh encoder for the live path. The test fixture helper `renderDeferredRayMarch` did not. **Live took the mesh branch; fixtures took the SDF branch.** Round 56's Lipschitz-corrected `sceneSDF` rendered clean cones in fixtures but didn't reach live because live wasn't using that code path — and I shipped six iterative tuning rounds (50–55) and a "clean fixture, real fix" celebration (round 56) without realising the gap. Round 57 retired the mesh path AND added a `useMeshPath: Bool` parameter to the fixture helper so future fixtures can verify whichever branch the live path uses. **Rule:** for any preset that has a dedicated GPU dispatch path different from the engine's default ray-march path, the fixture helper MUST exercise the same dispatch path the live app uses, by construction. Don't rely on "I think both paths produce the same output" — explicitly bind the same encoder, the same texture state, the same post-process chain. **Discriminator that should have caught this earlier:** Matt's live screenshots showing artifacts that my fixtures said were fixed. After 2 consecutive rounds of "fixture clean, live broken," the test/prod parity is the first thing to investigate, not yet another shader tuning pass. The cost of investigating gap was small. The cost of six wasted tuning rounds was large. Memory: `feedback_fixture_live_parity.md` covers this category broadly; the specific G-buffer-branch failure mode is the round-57 discovery.

67. **Routing the same audio timescale to two different visual layers** (Ferrofluid Ocean rounds 56–65, 2026-05-17 → 2026-05-18). Round 50 established the constant-field premise (spike geometry not audio-coupled). Rounds 56/60 tried to layer "subtle music response" back onto the spike heights via `bass_dev` — at the SAME TIMESCALE the swell amplitude was already being beat-coupled via `0.3 × drums_dev`. The two layers responding to similar beat-rate audio at different visual scales produced a "competing rhythms" reading: per-beat spike pulse AND per-beat swell pump moving simultaneously, encoding the same musical information twice through two different visual channels. Round 61 tried gating the spike pulse to `bar_phase01` (downbeat-only) — produced a different defect because bar boundaries don't align with bass kicks. Round 63 reverted to constant spikes. Round 65 finally landed the right configuration: REMOVE the drums coupling from swell amplitude (swell becomes arousal-only, slow), THEN reactivate per-beat spike-height response. With only one layer carrying the per-beat signal, the spike pulse reads as intended response without competition. **Rule:** when adding audio reactivity to a new visual layer, audit which other layers are responding to primitives at similar timescales. Don't route per-beat audio into two layers; don't route slow-arousal audio into two layers either. Each visual layer should consume one primitive at one characteristic timescale. **Architectural test for a new preset's audio-reactivity routing:** make a small table of (visual layer × audio primitive × timescale). If two layers share a primitive (or two primitives at the same timescale), that's the bug — the music will overdrive the same information through two visual channels and read as "fighting itself." Same lesson independent of preset: the architecture is one-primitive-per-layer.

69. **Iterating on automated cold-start beat-phase derivation from short live tap audio** (BUG-017, CS.1 → BSAudit.3.impl, 2026-05-22 → 2026-05-25). Six iterations attempted to derive the audible beat phase from the first ~3 s of live tap audio on a novel track: CS.1 baseline (trust cached grid; 7/10 fail ±50 ms / 90 %); CS.1.y.2 (sub-bass-onset snap — FA #68; 0/10, reverted); CS.1.y re-diagnosis (short-window Beat This! — 1-3/10 viable, non-reproducible); CS.1.y.2-redo r1 (Beat This!@15s snap — engine bug); r2 (Beat This!@15s snap — cross-capture-unstable, reverted); BSAudit.3.impl (BPM-prior + broadband-peak + confidence-gated accents — 4/10 pass on the new metric; perceptually iteration #6 of the same defect). Each iteration used a different mechanism; each failed in a different way. **The common thread is the upstream premise — "some automated signal in the first ~3 s reliably gives audible beat phase" — which six attempts have empirically falsified.** The failure modes the data exposed: sub-bass onsets fire on bassline events not beats; short-window Beat This! has unstable period estimation; Beat This!@15s is cross-capture-unstable; broadband flux fires on pre-beat content (pad swells, vocal entries, hi-hat lead-ins) so the BPM-prior anchors off-beat on 5/10 tracks, and confidence climbs anyway because periodic content at quarter-note rates reinforces *any* phase that matches the period prior. **Resolution (Matt's Choice A decision 2026-05-25):** initial closeout (`438edbbb`) retained BSAudit.3.impl as the production cold-start architecture and accepted the ±60 ms / 3 s perceptual sync sub-goal as structurally unachievable. **AMENDED 2026-05-26 — the impl was reverted on 2026-05-25 evening** (`33cd57e9` / `6758a617` / `002b5f2b` / `35305b5e`); production is the pre-impl baseline. The retired premise + the structural limit still hold; the gating runtime that the closeout described is no longer in the code. Diagnostic tooling (`--accent-window-pass-rate` verifier mode, SelfTest checks, diagnostic findings doc) was retained per Matt's "yes, keep the tools" sign-off. See CLAUDE.md §Cold-Start Phase Contract for the production-state description. **The discriminator:** any future "let me try yet another signal source for cold-start beat phase derivation" pitch needs a fundamentally different premise (human-tap reference per BSAudit-FU-5 Path B; full-track local-file analysis; per-track manual calibration UX) — not another short-window signal. Don't file iteration #7 without surfacing that premise change to Matt first. This is also the project-scope twin of Failed Approach #58 (Drift Motes — five failed iterations on the same preset defect) at infrastructure scope, with the same convergence rule: iteration converges only when each step integrates feedback into the model; six iterations without changing the upstream premise means the premise is wrong.

68. **Phase-locking the cold-start beat grid to live sub-bass onsets** (CS.1.y.2, BUG-017, 2026-05-22). The cold-start beat grid (Beat This! on the 30 s Spotify preview) has a reliable tempo but a per-track ±½-beat phase error. CS.1.y.2 tried to correct the phase by collecting the first live sub-bass onsets, taking the circular mean of their nearest-beat residuals, and applying a one-shot gross `drift` correction once a confident cluster formed (resultant `R ≥ 0.95`). It was engine-green (1272 tests) and reverted within the day: `ColdStartVerifier` scored **0/10, worse than the unfixed 3/10** — it regressed every previously-passing track by 100–300 ms. **Root cause: the sub-bass onset detector fires on sub-bass *events* (bass notes, 808s, synth bass), not *beats*.** On syncopated tracks those events are off-beat (Billie Jean −226 ms, Royals +316 ms), so the fix algebraically locked the visual onto the bassline (`visual = liveOnset`). The off-beat clusters are *tight* (MAD ~10 ms, dead-steady across the whole window), so they pass any tightness-based confidence gate — **a confidence gate on cluster tightness cannot tell an on-beat cluster from an off-beat one**. The discriminator that should have caught this earlier: the `LiveDriftValidation` love_rehab integration test regressed during implementation and I treated it as a tunable threshold problem (raised `R` 0.80 → 0.95) instead of recognising the input signal itself was wrong. **Rule:** the sub-bass onset detector is not a beat-phase reference — it is a sub-bass-*event* detector, and the steady-state EMA only survives it because its ±50 ms match window hard-rejects off-beat onsets (which is also why the EMA structurally cannot make a gross correction). The only reliable track-start beat-phase source is a real beat detector (Beat This!). Any future cold-start / beat-phase work must derive phase from Beat This! on live tap audio, never from raw onset-detector output. Cross-reference: this is the audio-layer twin of #4 (beat onsets are accent, never the primary driver) — onsets are not even a reliable *phase* reference, let alone a motion driver.

70. **Porting an external preset (Milkdrop/butterchurn) by patching Phosphene's pipeline one divergence at a time, instead of replicating its render loop wholesale from the source** (Dragon Bloom L4, D-138, 2026-06-02). The L4 "rich warm fill" took hours and a string of traded divergences (pale/washed → over-bright/flat → jittery → a GPU-stall hang) — not because the preset was hard, but because the work bent Phosphene's `mv_warp` (a *structurally different* feedback engine: separate scene texture, warp-with-decay, decayed compose, ping-pong) to imitate butterchurn's loop **incrementally**, fixing one symptom and exposing the next each round. The proof it was method, not difficulty: the moment the actual butterchurn source was read (`tools/dragon_bloom_reference/butterchurn.min.js`), every divergence resolved fast and consistently — custom-warp presets apply **no decay** (`warpColor=1`); feedback is **8-bit** (the clamp holds saturation; float over-accumulates pale); custom waves blend **normal-alpha on top** of the warped frame (that IS the feedback); comp (echo/gamma/invert) is **display-only**; symmetry is from the **video echo**, not strand mirroring; the warp is a **32×24 mesh**. **Rule:** when reproducing a system that has working source (an open-source engine, a reference implementation), replicate its *render loop / data flow as a coherent unit* by reading that source — do NOT approximate it piece-by-piece against your own differently-shaped pipeline and tune the gaps. The piece-by-piece path generates a long tail of divergences because each piece's correctness depends on the others being right. Tell-tale of the failure: you've "fixed" the same visual area 3+ times and each fix reveals a new wrongness in an adjacent area. This is the implementation-architecture twin of FA #64 (do desk research instead of first-principles guessing) and FA #49 (don't tune constants on a structurally-different renderer) — here the structural difference is the whole feedback loop, and the fix is to port the loop, not tune around it. The per-component butterchurn facts are durable in D-138; the discipline rule is this entry.

## Authoring Discipline — Process Failures to Not Repeat

These are operational rules distilled from the Drift Motes retirement (D-102, Failed Approach #58) and from prior preset iterations where Matt's feedback cycle was longer than necessary. They are *process* rules — how to work — not technical rules. Read them at the start of any preset session, before opening a `.metal` file.

**Strategy-scope clause (added DOC.3, 2026-05-13).** These rules apply at *strategy / product-commitment scope*, not just preset-authoring scope. Failed Approach #60 (Phase MD bloc) replayed the Drift Motes failure pattern (#58) at strategy scope — twenty decisions filed in one day, ten amended the same day, one reverted within 24 hours. When a strategic commitment is being drafted without empirical input from the work it governs, the same discipline rules fire: stop and bring the gap to Matt; do not produce structure as a substitute for an answer; do not treat "reusable infrastructure" as a defense for an unvalidated concept. A strategy decision is a forecast until evidence converts it; commit decisions when the work has produced evidence, not before.

**Articulate the musical role before authoring anything.** Before writing the JSON sidecar, before reading reference images, before opening a `.metal` file, write a one-sentence answer to: *"How is this preset's primary visual subject another instrument in the band?"* The sentence must name a specific musical feature (a beat, a downbeat, a sustained bass envelope, a vocal pitch contour, a structural boundary, a build-up) and a specific visual behaviour the listener will pair with it. "Vibe with the music" / "reactive to energy" / "feels like the song" are not answers. If you cannot produce the sentence, stop. Bring the gap to Matt before scoping the increment. The Drift Motes concept failed this gate and four days of iteration did not recover.

**Reference images are still moments; presets are behaviours over time.** A photograph in `docs/VISUAL_REFERENCES/<preset>/` shows what the *frame* should resemble. It does not encode the *temporal contract* — what changes when the bass drops, what fires on a downbeat, what carries the listener through a 30-second cycle. Failed Approach #39 (authoring without reference images) is the floor; meeting it is not enough. Pair every reference image with an explicit temporal contract before the first commit.

**Three-part bar for any new preset concept.** Every concept must clear all three:
1. **Iconic visual subject deliverable at fidelity** — you can demonstrate, from a comparable past preset, that the visual style is reachable. Honest self-assessment: if Matt has flagged a fidelity gap on a similar preset before, your default is "I cannot deliver this" until proven otherwise.
2. **Clear musical role** — see the previous rule.
3. **Infrastructure-feasible** — does not require render passes / engine surfaces / GPU contracts Phosphene lacks. If the answer is "we can add the infrastructure," check whether Matt agrees the infrastructure is worth the increment.

Pitches that pass two of three and require the user to spot the missing third are not acceptable. Surface concerns *before* the pitch, not when cornered.

**The next response to pushback must change the answer, not justify it.** When Matt flags a problem, the failure mode I fall into is producing structured analysis — "I can see X about the music," pros/cons lists, three-part frames — as a substitute for thinking. The user perceives this correctly as bullshit-generation: *"Everything you are telling me is bullshit"* / *"Stop pitching"* (2026-05-11). If a reply has more structure than substance, delete the structure. Real thinking changes the answer; decorating the existing answer with more framework is cope.

**"Reusable infrastructure" is not a defense for a failed concept.** When a preset's concept doesn't work, the implementation does not earn preservation as "kernel waiting for the right concept" or "scaffolding for a future preset." D-097 (siblings, not subclasses) means future particle presets ship their own conformer and shader file. The Drift Motes `motes_update` kernel was specific (force-field motion + age recycle + per-particle hue bake) — it could not host an unrelated concept without rewrite. If "infrastructure" appears in a defense of keeping deleted-concept code, the defense is wrong.

**Iteration converges only when each step integrates feedback into the model.** A remediation increment that ships green but doesn't update your understanding of why the preset is failing is not progress. After every M7 round, write one sentence: "what I now believe about why this preset is failing." If the sentence doesn't change between rounds, the iteration is mechanical and the next increment is wasted. Stop and re-scope.

**Treat Matt's fidelity warnings as constraints, not discussion points.** If Matt says, on the basis of observed past sessions, that you will not be able to deliver a particular fidelity target — treat that as a binding constraint. Pitch only within the achievable bar, or say "I cannot deliver this" upfront. *"B or C will be botched by you because you won't be able to achieve the level of visual fidelity needed. I have literally watched this happen for the last several preset designs"* (2026-05-11) is data, not a debate prompt.

**Verify against the artifact before asserting facts about it.** Several smaller failures in the Drift Motes timeline came from confident-but-wrong assertions: a perf-capture procedure that cited a non-existent CSV column, a misattribution of a test failure to V.7.7C.4 when working-tree state showed V.7.7C.5, a "what's next" pitch for BUG-011 work that `git log` would have shown the parallel session was actively doing. The verification cost (one grep, one `git log`, one file read) is always less than the cost of being corrected. Default to verification when the claim is checkable.

**Decisions presented to Matt must be framed in product-level language with explicit benefits and trade-offs.** Matt is product/design lead, not a peer engineer. Any decision presented to him for input must:

1. **Be framed in user-visible terms, not engineering jargon.** Use "how dense are the spikes" not "particle count = 256 / 512 / 1024." Use "how responsive is the motion" not "update tick frequency." Use "how sharp are the peaks" not "smooth-min `w` parameter."
2. **Call out benefits and trade-offs in plain English.** For each option, name what the user sees / feels — not what the code does. E.g. "Medium-dense (peaks touch base-to-base; surface reads as covered in ferrofluid)" vs "Densely-packed (individual peaks blur into continuous textured field; closest to references but more processing overhead)."
3. **Include a recommendation with reasoning.** Default value Matt can accept if he has no strong preference; he should not have to do engineering math to ratify your default.
4. **Limit the question to decisions that actually have product-level consequences** (visual character, motion feel, audible behaviour, UX flow). Engineering implementation choices (CPU vs GPU, buffer sizes, math precision, internal data structures) are Claude's responsibility — make them yourself, explain only the user-facing consequence if any.

If a question requires Matt to already know implementation details to answer it, the question is wrong — reframe at the product level or make the decision yourself.

**Tell-tale phrasing of the failure:** asking Matt "Option (a): 256×256, Option (b): 512×512, Option (c): 1024×1024" — three values that mean nothing without engineering context. Reframe as "How sharp should the peaks render? Adequate (1.5 cm per pixel) / sharp / very sharp" with what the user sees at each, recommendation, and a default.

**Discovered:** V.9 Session 4.5b draft prompt (2026-05-14) asked Matt to choose particle count, height-map resolution, smooth-min parameter, and update location — four engineering values requiring implementation context to evaluate. Matt called it out: *"you keep insisting on communicating with me like I'm a peer engineer."* The rule above was promoted to CLAUDE.md immediately as a binding discipline rule.

**Escalation thresholds.** Stop and bring the gap to Matt — do not start the next increment — when any of these fire:
- Two consecutive M7 reviews return negative feedback whose root cause you cannot articulate.
- A successor-concept pitch fails the three-part bar.
- You catch yourself producing structure as a substitute for an answer.
- A "reusable infrastructure" argument is forming in your response.
- The one-sentence "what I now believe about why this preset is failing" hasn't changed between rounds.

The cost of pausing is small. The cost of another day of mechanical iteration on a broken concept is high.

**Test in the production-grade rendering pipeline. No shortcuts.** Promoted to CLAUDE.md from Matt's instruction 2026-05-18 after AV.1 / AV.2 / AV.2.1 each shipped with green tests despite producing painterly smear in live playback — the Aurora Veil test harnesses rendered single frames through `preset.pipelineState` directly, bypassing the mv_warp accumulator the live app uses. Three increments wasted before the diagnostic test that exercises mv_warp was written.

**The rule:** every preset increment that depends on temporal behavior must include a test that runs the same dispatch path the live app uses — for ≥ N frames where N covers the relevant accumulator decay window — and inspects the multi-frame output. "Same dispatch path" means: if the preset uses `mv_warp`, the test runs scene → warp → compose → swap in a loop. If it uses `staged`, the test runs all stages. If it uses `ray_march`, the test runs G-buffer → lighting → composite. Tests that bypass to `preset.pipelineState` alone are sufficient ONLY for verifying the shader's instantaneous output; they do NOT prove the preset works under the live rendering pipeline.

**Concrete obligations:**
1. **Before authoring a preset that uses `mv_warp` / `staged` / `feedback` / `ray_march` + `post_process`** — write or extend the multi-frame test harness FIRST. Verify the live pipeline path is reachable from a test before shipping any shader work.
2. **For mv_warp-using presets specifically** — `AuroraVeilMVWarpAccumulationTest` is the reference pattern (env-gated, runs scene → warp → compose → swap for 60 frames at silence, captures the final accumulator + a quantitative star-count metric). Adapt for new presets; do not reinvent.
3. **For staged presets** — the harness must bind the same slot-6 / slot-7 buffers the production path binds (CLAUDE.md FA #66; same principle generalizes).
4. **Single-frame tests stay valuable** for shader-math gates (palette stratification, deviation primitives, brightness amplitude). They are NOT sufficient on their own.
5. **Closeout reports must state which dispatch path the tests exercised.** "Tests pass" alone is no longer evidence of correctness.

**Tell-tale phrasing of the failure:** "the silence test passes, so silence is stable" / "the visual review PNG looks clean" / "the regression hash is within Hamming threshold." None of these prove the preset works under the live pipeline if the test bypasses it. Surface this and write the missing harness BEFORE filing the increment as ✅.

**Design is upstream of testing — surface risks immediately.** A passing test of a flawed design just ships the flaw faster. When the design doc proposes a mechanism without empirical grounding (e.g., AV's "mv_warp on top of nimitz's procedural recipe" — no published aurora demo combines those), name that as a risk to Matt **before** writing any code, not after the rendering smears in live.

**Grounding priority (Matt-approved 2026-05-18, soft rule).** Required grounding for each mechanism, in descending order of preference:
1. **Working code reference in a comparable visual context.** Cite Shadertoy ID, GitHub repo, demo URL.
2. **Academic paper or physics derivation + clear math** that can be implemented from the description alone. Example: Lawlor's `H(z)` curve from the WSCG 2011 paper.
3. **No reference, just the design doc's assertion.** Highest risk. Surface explicitly as "no empirical grounding for X" before authoring; Matt decides whether to accept the risk or descope.

When combining two mechanisms (e.g., procedural recipe + feedback accumulator), the combination itself needs grounding — not just each piece individually. AV's failure was layering mv_warp on top of nimitz without any working aurora demo doing the same combination; neither piece alone was the problem, the combination was.

The threshold for "surface immediately" is when you reach grounding level 3 (no reference) on any non-trivial mechanism. Don't proceed silently on assertion alone.

**Diagnostic infrastructure precedes fidelity claims.** Promoted to CLAUDE.md from the AV.2.x cascade closeout 2026-05-20. A closeout that asserts an audio-coupled route works must cite per-route firing evidence from the session's `features.csv` / `stems.csv` — frame counts, threshold-crossing percentages, video-frame extracts at the audio events. A closeout that asserts the rendered output belongs in the same visual family as the references must cite the per-question rubric proxy scores + reference family centroid + σ-distance verdict. "Visually verified," "reads in the same visual conversation as references," and "the route works" are gate-bypass language unless backed by cited evidence.

**The tool that produces the evidence is `PresetSessionReplay` (SR.1, 2026-05-20).** See `docs/ENGINE/SESSION_REPLAY.md` for invocation + extension. Every preset closeout asserting audio-coupling or visual-fidelity claims runs the harness against the relevant session + reference set and embeds (or links) the generated `replay_report.md` in the closeout.

**When the diagnostic doesn't exist for a question, the closeout says "cannot verify X" instead of asserting it. Building the missing diagnostic is the next increment, not a future task.** PT.1 was the existence proof: `vocalsPitchConfidence` was 0 % across every Aurora Veil session for ~5 months while closeout after closeout I authored claimed the route worked. The 10-line script tallying `vocalsPitchConfidence ≥ 0.5` from features.csv that would have caught it before AV.1 shipped didn't exist; I filled the gap with assertion-shaped language instead of building it. That pattern is retired.

**Verdicts on broken proxies are forbidden.** SR.1's rubric calibration flags a proxy `uncalibrated` when its scores across the reference set are too scattered (σ > 50 % of |mean|) or constant (σ ≈ 0). "Uncalibrated" is an honest verdict — it means the proxy isn't reliable, NOT that the render is OK. A closeout that cites an uncalibrated proxy as evidence of cert-readiness is gate-bypass. Either refine the proxy (SR.1.x / SR.2) or say "cannot verify."

## What NOT To Do

- Do not block the render loop on network, ML, or metadata.
- Do not allocate in the Core Audio IO proc callback.
- Do not use `.storageModeManaged` buffers.
- Do not make beat onset the primary visual driver.
- Do not hardcode shader paths.
- Do not normalize 6-band AGC per-band.
- Do not pass `SWIFT_TREAT_WARNINGS_AS_ERRORS=YES` on the xcodebuild command line.
- Do not assume Now Playing metadata is available or accurate.
- Do not use `[[thread_index_in_mesh]]` in MSL (does not exist).
- Do not deform architecture geometry with audio in ray-march scene presets — modulate light/atmosphere/camera instead (DECISIONS D-020).

- Do not ship pale-dominant panels. Cream / ivory / pearl / pale-anything tones (linear RGB with `min(R, G, B) > 0.65`) are **permitted as structural highlight** against saturated or deep grounds — stained-glass, porcelain, miniature, Cycladic palettes all use them this way. They are **forbidden as dominant ground**: a panel where pale cells exceed 30 % of the dominant area is rejected. The pale-tone-share ceiling (≤ 0.30) is the mechanical enforcement at LM.9 (and equivalent per-preset cert gates for any future palette-driven preset). Mood-tint formulas of the form `mix(cream, hue, sat)` that pull every cell toward a desaturated baseline regardless of mood input remain an anti-pattern in shader authoring — they produced the rejected LM.2 Lumen Mosaic output and the LM.4.5 v1 pastel guardrail. Use procedural palettes (V.3 `palette()`), the LM.4.7 curated palette library (D-LM-palette-library), or per-stem saturated colour constants — and audit aggregate pale-share, not just per-cell colour. Quiet *moments* (silence, intros) are allowed; the active visual register stays vivid. Supersedes the prior categorical "no muted / pastel / cream-haze palettes" rule per D-LM-cream-rescission. Phosphene is still vivid by project register; the rule change is compositional, not about loosening saturation discipline.
- Do not write `smoothstep(0.x, 0.y, fbm8(...))` with thresholds above 0.3. `fbm8` is centred at 0 with practical range `[-0.7, 0.7]` — thresholds in the `[0.4, 1.0]` band will always return 0 and produce no spatial variation. Centre thresholds near 0 or remap first (`v * 0.5 + 0.5`). Same for `worley_fbm` (effective range `[-0.65, 0.79]`). Failed Approaches #42, #43.
- Do not sample `fbm8(wp)` at scale 1.0 on unit-sphere or unit-cube positions. Perlin lattice-point degeneracy kills variation (measured std dev ≈ 0.05 instead of expected 0.4). Use scale ≥ 3, preferably 5–10. Failed Approach #43.
- Do not use Metal built-in type names as local variable names in `.metal` files. `int half = spokeCount / 2` shadows the `half` float type, causing a silent compilation failure — the shader is silently dropped from the preset fixture with no stderr output, and regression tests pass trivially because no golden hash is ever checked. Avoid `half`, `ushort`, `uchar`, `packed_float3`, and other Metal type keywords as variable names. Failed Approach #44.
- Do not assume Spotify API response schemas are stable across endpoint renames. When `/tracks` became `/items`, the track key changed from `"track"` to `"item"`. Always consult current Spotify Web API reference docs before implementing connector parsing. Failed Approach #45.
- Do not use Spotify's `fields` query parameter on the `/items` endpoint. It silently returns `{}` for items where any requested field is null, producing an empty track list with a 200 response and no error. Failed Approach #46.
- Do not discard `preview_url` from a Spotify API response and then query iTunes Search to find it again. The Spotify `/items` response already contains the CDN preview URL per track. Capture it in `TrackIdentity.spotifyPreviewURL` and short-circuit `PreviewResolver`. Failed Approach #47.
- Do not fuse onset events across bands (sub_bass + low_bass) when feeding the IOI tempo histogram. Per-band cooldowns are independent and FFT-hop quantization produces alternating-frame staggers that bias BPM estimation toward faster buckets. Source from `result.onsets[0]` (sub_bass per-band events from `detectOnsets`) only — D-075, Failed Approach #50.
- Do not re-introduce histogram-mode picking on integer-rounded BPM buckets. Bucket widths grow with BPM in period space, biasing the mode toward faster BPMs. Use trimmed-mean IOI (`computeRobustBPM`) — D-075, Failed Approach #51.
- Do not write the literal `44100` in any code path that should consume the live tap rate. Capture `tapSampleRate` immutably at tap install and thread it through every consumer (stem separator dispatch, per-frame stem analysis, `StemSampleBuffer`, `StemAnalyzer` init, live Beat This! `analyzeBeatGrid`, `SessionRecorder.recordStemSeparation`). The literal is allowlisted only in `StemSeparator.modelSampleRate`, `BeatThisPreprocessor.sourceSampleRate`, and test fixtures — `Scripts/check_sample_rate_literals.sh` enforces this in CI. D-079, Failed Approach #52.
- Do not mutate `tapSampleRate` from the audio thread without a synchronization barrier. Cross-core visibility is not guaranteed for unsynchronized 8-byte writes on Apple Silicon. Capture once per `installTap(...)` (`let` post-install) or guard with `os_unfair_lock`. If capture-mode switching changes the rate, tear down and re-init dependent buffers (`StemSampleBuffer`, `StemAnalyzer`) on the rate change rather than mutating the rate field in place. D-079.
- Do not double sub-80 BPM in any tempo path. `BeatGrid.halvingOctaveCorrected` is halving-only because Pyramid Song genuinely runs at ~68 BPM and other tracks in [40, 80) BPM exist. `BeatDetector+Tempo.computeRobustBPM` and `estimateTempo` must match — drop any `if bpm < 80 { bpm *= 2 }` branch. D-079, post-QR.1.
- Do not store `MIRPipeline.elapsedSeconds` as `Float`. After 30 minutes of `+= deltaTime` accumulation, ULP ≈ 240 µs — a guaranteed monotonic drift smaller than the ±30 ms tight-match window but corrosive over a long session. Store as `Double`; convert once at FeatureVector write. Same rule for any other long-accumulator wall clock.
- Do not sum AGC-normalized stem energies for orchestrator scoring. AGC centers each stem at 0.5; summing 2+ saturates trivially and the 25% score-weight slot does no work. Use deviation primitives (`stemEnergyDev`, D-026) and take the *mean* over declared affinities, not the clamped sum. D-080, Failed Approach #53.
- Do not score reactive-mode tracks against `TrackProfile.empty` with `stemEnergyBalance == .zero` and a stem-affinity-aware scorer. The combination adversarially rejects the most musically-engaged catalog presets. Score against neutral 0.5 until the live stem analyzer has converged (~10 s), then against a live `StemFeatures` snapshot. D-080, Failed Approach #54.
- Do not call `applyLiveUpdate` (mood-override path) without a per-track cooldown. The path runs at ~94 Hz on the analysis queue; without rate-limiting it re-patches `livePlan` on every frame the conditions hold. 30 s per-track cooldown is the documented minimum. Boundary reschedule is unaffected (it has its own gate).
- Do not instantiate a second `SettingsStore` anywhere in the app. There is one app-wide instance constructed in `PhospheneApp.swift` and injected via `@EnvironmentObject`. A `@StateObject private var settingsStore = SettingsStore()` inside any view creates a parallel state world that user toggles never reach (Phosphene reviewer 2026-05-06: `PlaybackView` had a duplicate that silently swallowed every capture-mode change). `SettingsStoreEnvironmentRegressionTests` enforces this.
- Do not match plan entries against the live track via lowercased title+artist string. Covers, remasters, and encoding differences break the match. Publish `currentTrackIndex: Int?` from `VisualizerEngine` (the plan walk already knows the index) and bind to that.
- Do not write a feature's `@Published` surface on one code path without clearing it on the complementary path. A publisher retains its last value indefinitely; if a feature populates it on path A (e.g. LF track-change) but a parallel mode (e.g. streaming track-change, or session-boundary) never writes it, the stale value leaks across the mode transition. LF.6's `currentTrackArtworkData` was written on the LF path but not the streaming path → the prior LF session's album art rendered against every streaming track (BUG-024). When you add a `@Published` that a feature writes, enumerate every path that changes the underlying context (every track-change callback, every `.connecting`/`.ended` session-boundary observer) and write-or-clear it on each. Pair the clear in the same MainActor tick as the sibling publisher (e.g. title + artwork together) so consumers never observe a half-updated surface.
- Do not silently skip a test on a missing fixture. `Issue.record(...)` or `XCTFail(...)` instead — silent skips have already disappeared the entire DSP.2 S8 four-bug regression surface on fresh checkouts at least once. `BeatThisFixturePresenceGate` enforces fixture presence as a precondition.
- Do not call `drawWorld()` from `arachne_composite_fragment`. The WORLD stage owns it; COMPOSITE samples the resulting texture at `[[texture(13)]]` via `worldTex.sample(arachne_world_sampler, uv)`. Re-rendering WORLD in COMPOSITE defeats the staged-composition contract V.ENGINE.1 / V.7.7B established and pays the WORLD render cost twice per frame. Same rule applies to any future staged preset: cross-stage geometry composites by texture sample, not by inline recomputation. The rule extends to drop refraction blocks: drops sample `worldTex` at `[[texture(13)]]` (V.7.7C §5.8 recipe), never inline `drawWorld(refractedUV, ...)` per-drop-pixel. The dead-code reference `drawBackgroundWeb()` in `Arachne.metal` ~line 563 calls `drawWorld()` directly because it predates the stage split — V.7.7C does NOT regress that pattern; future foreground recipes follow the worldTex sample form. D-092 / D-093.
- Do not bind anything other than the per-preset fragment buffers (`directPresetFragmentBuffer` / `directPresetFragmentBuffer2`) at fragment slots 6 / 7 in the staged or mv_warp paths. Slots 6 / 7 are CLAUDE.md GPU-Contract reservations — Gossamer's wave pool, Arachne's web pool + spider GPU struct, and any future per-preset world state all flow through these slots via the existing setter API. Other presets needing additional buffers must extend `RenderPipeline` with `directPresetFragmentBuffer3` / `4` (and matching staged-encode binding) rather than overloading the reserved slots. D-092.
- Do not expand `ArachneSpiderGPU` past 80 bytes. Listening-pose and gait state stay CPU-side; the GPU contract is `tip[8]` only. Adding fields like `listenLift: Float` would push the struct to 96 bytes (16-byte alignment), break the slot-7 buffer allocation in `ArachneState.swift:204` (`spiderBufSize = MemoryLayout<ArachneSpiderGPU>.stride`), and force a regression of every spider golden hash. The V.7.7D listening pose realises the lift by adjusting `tip[0]` / `tip[1]` clip-space Y in `writeSpiderToGPU()` before the GPU bind — the shader's IK derives the raised knee analytically from the lifted tip without any extra channel. Same rule extends to other per-preset world state at slot 6 / 7. D-094.
- Do not call `mat_chitin` (V.3 cookbook) with the V.3 default blend (`thin × 1.0`) in any orb-weaver context. §6.2 mandates the thin-film iridescence ride at biological strength `× 0.15` (≤ 0.20 invariant) — higher reads as the §6.2 anti-reference (ref `10` neon glow, Failed Approach #48). The V.7.7D Arachne spider path inlines the §6.2 recipe (`base + thin × 0.15 + fuzz + bodyLit + rim`) and bypasses `mat_chitin` entirely; the cookbook entry stays in `Materials/Organic.metal` for other contexts where the V.3 default blend is intended (close-up insect-cuticle hero shot, etc.). If a future preset uses `mat_chitin` in an orb-weaver context, multiply the resulting emission by `0.15` before adding to the surface colour. D-094.
- Do not implement Arachne vibration in WORLD or via per-vertex deformation. WORLD is intentionally still per §8.2 anchor-vs-tip physics — silk shakes against a stationary forest. Arachne is a fullscreen-fragment preset; per-vertex deformation would require a vertex stage which Arachne does not have. Vibration is a fragment-space UV jitter on COMPOSITE web walks (`arachneEvalWeb(vibUV, ...)`) + spider body translation only. The bottom-of-fragment `worldTex.sample(...)` keeps the original `uv`. D-094.
- Do not advance Arachne build stages in beats. Build pace is audio-modulated TIME (`dt × pace`, where `pace = 1.0 + 0.18 × midAttRel + max(0, 0.5 × drumsEnergyDev)`). V.7.5's beat-measured stage timing was admitted in `ARACHNE_V8_DESIGN.md §5.2` to be wrong (inconsistent build cadence across tracks). The pause guard MUST be evaluated BEFORE `effectiveDt` is computed — if you compute `dt × pace` first and then zero it on spider activity, the spider blend ramp time bleeds into the build timeline. D-095. **BUG-011 round 8 (2026-05-12)** added a second guard alongside the spider-pause: `effectiveDt = 0` when the sum of the four AGC-normalised stem energies < 0.02 (silent-state pause — prep / silence / source-app paused). Same ordering rule applies: set `pausedBySpider` from `spiderBlend > 0.01` BEFORE evaluating the silence gate, otherwise the spider-pause regression test (which uses `stems: .zero`) will trip on the silence gate without latching `pausedBySpider`. Both gates collapse `effectiveDt` to 0; only the spider gate latches the flag.
- Do not transition `wait_for_completion_event: true` presets on duration or section-boundary timers mid-build. **BUG-011 round 8 (2026-05-12)**: this flag (currently set only on Arachne) means the preset's contract is "let me finish before you transition me." `maxDuration(forSection:)` returns `.infinity` for these presets (same path as `isDiagnostic`), so the V.7.6.C formula's `motion_intensity / fatigue / linger` penalties and the `naturalCycleSeconds` cap are both bypassed. Mood-derived `presetOverride` is stripped in `applyLiveUpdate` for the active segment (mirroring `diagnosticPresetLocked` and `isCaptureModeSwitchGraceActive`). The runtime transition trigger is the preset's `PresetSignaling.presetCompletionEvent`, wired through the existing `wirePresetCompletionSubscription` path. The `remainingInSection` cap in `planOneSegment` is intentionally still active — section boundaries are the only hard time bound on completion-gated segments, accepted because typical sections are ≥ 60 s and Arachne's round-8 build cycle is ~92 s. If you find yourself adding a new transition path that doesn't honour the `waitForCompletionEvent` flag, surface the scope with Matt before landing it.
- Do not add the V.7.5 4-web pool back as the foreground rendering source. The post-V.7.7C.2 composition is ONE foreground hero web (driven by `webs[0]` Row 5 BuildState) + 1–2 saturated background webs at depth (V.7.5 spawn/eviction continues to run for `webs[1..3]` as background depth context only). Spawn / eviction logic was retired for the FOREGROUND; if you find yourself making the foreground anchor block read `webs[0].stage` / `webs[0].progress` (Row 2) instead of Row 5 fields, stop — that's V.7.5 thinking. The shader's pool loop starts at `wi = 1` for the same reason. D-095.
- Do not call `arachneState.reset()` outside `applyPreset`'s `.staged` branch. The build state machine's segment start is the canonical reset point; ad-hoc resets break per-segment spider cooldown semantics (`spiderFiredInSegment` would clear at the wrong time, allowing the spider to re-trigger mid-segment). The visual-review harness (`PresetVisualReviewTests`) deliberately skips `reset()` because it doesn't simulate orchestrator segment boundaries — that's why the harness renders the foreground in frame phase at low progress. D-095.
- Do not gate the chord spiral on per-ring visibility. Per-ring `kVis = (k / N_RINGS) <= progress` makes an entire ring's chord segments + drops appear at once as a complete oval — Matt's 2026-05-08T17-01-15Z manual smoke flagged this as the §5.6 "spider lays chords one-at-a-time" violation. Use the V.7.7C.3 per-chord gate `globalChordIdx < int(progress × N_RINGS × nSpk)` so chords reveal outside-in by ring, clockwise-by-spoke within. D-095.
- Do not re-enable the V.7.5 pool web rendering loop. Matt's 2026-05-08 manual smoke flagged the V.7.5 spawn/eviction churn ("full webs flash on and fade away ... new webs form over the central web being spun") as competing with the foreground build, not framing it. V.7.7C.3 changed the shader's pool loop bound from `wi < kArachWebs` to `wi < 1` (empty body) and the foreground hero is now the only rendered web for Arachne. CPU-side V.7.5 spawn state still advances harmlessly so unit tests pass; if you find yourself raising the bound back to `kArachWebs`, stop. The §5.12 1–2 saturated background webs need a separate slot-8 buffer flush for `ArachneBackgroundWeb`, NOT a re-enabled V.7.5 pool. D-095.
- Do not gate the spider trigger on `subBass + bassAttackRatio < 0.55`. Live LTYL session data (2026-05-08T17-01-15Z) confirmed this combination is acoustically impossible on real music: kicks have `subBass > 0.30` but `bassAttackRatio > 1.0` (sharp transient against AGC); sustained sub-bass passages have `subBass` near AGC average. The 0.75 s sustain accumulator decays at 2× rate during sub-threshold frames so it never reaches threshold on either pattern. V.7.7C.3 trigger is `features.bassAttRel > 0.30` (smoothed bass envelope — same primitive the §8.2 vibration uses successfully); brief kick pulses are filtered by the existing 0.75 s sustain accumulator unaided. Failed Approach #57. D-095.
- Do not pack per-segment Arachne polygon state into `webs[0]` Row 5 or a new WebGPU row. Row 5 fields (`build_stage` / `frame_progress` / `radial_packed` / `spiral_packed`) are the V.7.7C.2 BuildState contract; adding more fields would push the struct stride past 96 bytes. V.7.7C.3 packs the polygon anchor indices into the existing `webs[0].rngSeed` field (UInt32, byte offset 28) via `ArachneState.packPolygonAnchors(_:)` — the V.7.5 spawn driver no longer consumes that field on the shader side after Fix 2 retired pool rendering. If you need additional per-foreground state beyond what fits in `rngSeed` (32 bits) and Row 5, allocate a side buffer at slot 8 instead. D-095.
- Do not parameterize `ProceduralGeometry` to host a non-Murmuration particle preset's behavior. Particle presets are siblings, not subclasses (D-097). The `["feedback", "particles"]` pass is dispatched through the `ParticleGeometry` protocol (`update`, `render`, `activeParticleFraction`); each preset ships its own conformer + its own `Particles*.metal` engine-library shader file rather than parameterizing `ProceduralGeometry`'s kernel-name lookup, particle count, drag, or `decayRate: 0` flock-persistence semantics. If you find yourself adding a new field to `ParticleConfiguration` because Murmuration doesn't use it, stop — that field belongs in a new conformer's private state. (Drift Motes, the only other particle preset that ever shipped, was retired in D-102; future particle presets do not branch from its deleted code.)
- Do not extend an MSL struct in `Common.metal` (or any engine library shared file) without verifying byte-identical reads via `PresetRegressionTests` golden-hash regen across every preset. The DM.2 `FeatureVector` 32 → 48-float and `StemFeatures` 16 → 64-float extension (D-099) was safe because the first 32 / 16 floats kept their original offsets — Murmuration's `particle_update`, MVWarp's vertex/fragment, and `feedback_warp_fragment` all read fields in the original tail. If a future struct edit reorders existing fields or inserts a new field anywhere in the original tail, every engine-library reader's byte access shifts and silent visual regressions land. The Swift-side `CommonLayoutTest` regression-locks the sizes against shrinkage; the golden-hash sweep is the load-bearing visual check. D-099. (Drift Motes was the original consumer of the extended fields; after its removal in D-102, no engine kernel reads past the original 32 / 16 floats, but the extended layout is preserved so a future engine kernel can read MV-1 / MV-3 fields without forcing another size migration.)
