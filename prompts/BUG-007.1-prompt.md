────────────────────────────────────────
SYMPTOM
────────────────────────────────────────
Once `LiveBeatDriftTracker.computeLockState()` returns `.locked`, the
mode label `● PLANNED · LOCKED` should hold for the rest of the
track on stable input. In practice it oscillates between
`◑ PLANNED · LOCKING` and `● PLANNED · LOCKED`, holding LOCKED for
only a few seconds at a time. After ~25–30 s of playback,
`drift_ms` in `features.csv` plateaus at a fixed value and stops
moving — onsets have stopped matching at all.

The plateau pattern is the load-bearing tell. Two captures show it:

  1. Love Rehab (Test 4, ad-hoc reactive after live Beat This!
     installed grid at bpm=125.8). drift_ms walks linearly negative,
     pegs at exactly **−90.490 ms** (the ±90 ms onset-search-window
     edge — `Self.driftSearchWindow` is 0.05, but the actual peg
     value suggests a different cap is in play; verify which path
     produces this). Onsets stop matching; lock_state never holds 2.

  2. **Money 7/4 (BUG-006.2 capture, prepared grid bpm=123.2)**.
     drift_ms settles at +5 to +17 ms — well inside the ±30 ms
     `strictMatchWindow` — for ~25 s, yet lock_state flips 1↔2
     throughout that window without ever holding 2. After 28 s
     drift pegs at **+14.396 ms** and stops moving entirely. **This
     is the cleanest evidence** because the prepared grid is
     correct (123.2 is within 1.4 % of 125 — far inside any sensible
     tolerance), so the failure is purely in the lock hysteresis,
     not the BPM. The grid is right; the tracker still fails.

The Love Rehab plateau is corrupted by BUG-008's BPM disagreement
(prepared grid 118.1 BPM vs. live tap at 125 BPM — drift_ms can't
help walking out of the search window). The Money plateau is not.
**Diagnose against Money primarily**; Love Rehab is a secondary
data source that confirms the same plateau pattern but with extra
confounding noise.

────────────────────────────────────────
LOAD-BEARING CONTEXT
────────────────────────────────────────
LiveBeatDriftTracker is at
`PhospheneEngine/Sources/DSP/LiveBeatDriftTracker.swift`. The lock
state machine is small enough to read end-to-end. Key constants
(documented as of 2026-05-06):

  - `driftSearchWindow = 0.050`            (±50 ms onset → beat match)
  - `onsetAlpha = 0.4`                     (per-onset EMA blend)
  - `decayTau = 1.0`                       (no-onset drift decay τ)
  - `strictMatchWindow = 0.030`            (±30 ms tight-match for lock counter)
  - `lockThreshold = 4`                    (matched onsets needed to reach LOCKED)
  - `lockReleaseMisses = 3`                (consecutive misses to drop LOCKED)

Update path (lines ~210–290 of the file):

  1. For each onset, find nearest beat in grid via `nearestBeat(to:within:)`
     using `driftSearchWindow`. If no beat within ±50 ms,
     `consecutiveMisses += 1` and return.
  2. Compute `instantDrift = onsetTime − nearestBeat.time`.
  3. If `|instantDrift − drift| < strictMatchWindow`,
     `matchedOnsets += 1; consecutiveMisses = 0`. Else
     `consecutiveMisses += 1`.
  4. Update `drift = drift + onsetAlpha * (instantDrift − drift)`.
  5. `computeLockState()`: returns `.locked` iff
     `matchedOnsets ≥ lockThreshold && consecutiveMisses < lockReleaseMisses`.
     Otherwise `.locking` (if any history) or `.unlocked`.

The plateau pattern likely emerges from this combination:

  - `consecutiveMisses` increments on every "no nearest beat in ±50 ms"
    event but is reset only on a strict-match (±30 ms). On dense mixes
    where onsets are noisy and frequently land in the 30–50 ms band,
    `consecutiveMisses` accumulates while `matchedOnsets` doesn't.
  - Once the search-window check fails, the EMA freezes (`drift` is
    only updated when there is a `nearestBeat`). Consecutive
    no-nearest-beat events leave `drift` static while the live audio
    continues to drift.
  - On re-entry (an onset finally lands within ±50 ms again), the
    instant drift may be far from `drift` (which froze 5 s ago), so
    `|instantDrift − drift| > strictMatchWindow` → still no match.
    Now we have a chicken-and-egg: drift can't update because
    matches are rejected, and matches are rejected because drift is
    stale.

This is a **plausible** hypothesis. Diagnosis must verify which
specific code path produces the plateau on Money, with numbers from
the actual session.

There is no decay on `matchedOnsets` itself — once you reach
`lockThreshold = 4`, the only way to drop below is for
`consecutiveMisses ≥ 3`. So flickering 1↔2 means
`consecutiveMisses` is bouncing across 3 repeatedly while
`matchedOnsets ≥ 4` stays put. Look for that pattern in the
diagnosis output.

DIAGNOSIS PATH
────────────────────────────────────────
Three orthogonal checks. Run all three; cross-reference the answers
in the writeup.

**Check 1 — Replay Money's live onsets against its prepared grid
in a test, with full lock-state tracing.** This is the load-bearing
check.

The Money capture has `raw_tap.wav`, `features.csv` (with onset
events column if you can find it — verify by reading the file), and
`session.log` showing the prepared grid was installed at
`bpm=123.2, meter=2/X`. Either:

  (a) reconstruct the onset times from `features.csv`'s
      onset/flux columns, or
  (b) re-run BeatDetector against `raw_tap.wav` to produce the
      same onset stream the live tracker saw,
  (c) write a deterministic test fixture: assert grid.beats based
      on the captured prepared grid (123.2 BPM, 4/4 from the
      session log), and inject a synthesized onset stream with
      jitter that mirrors what real-music kicks produce
      (±10–25 ms typical), at 60 s duration.

Whichever route is faster — instrument LiveBeatDriftTracker (or
add a diagnostic mode that captures every update step into a
ring buffer) and replay Money's audio. Print on every onset:

  t=<sec> onset_t=<sec> nearest_beat=<sec or nil> instant_drift=<ms>
  prev_drift=<ms> matched=<int> misses=<int> lock=<.locked/.locking/.unlocked>

The plateau will be obvious in this trace — at t=28 the prints
will show "nearest_beat=nil" or "instant_drift far from
prev_drift" repeatedly, and `consecutiveMisses` will climb past 3.

The deliverable from Check 1: identify whether the failure is
(a) `nearestBeat` returning nil (onsets falling outside ±50 ms),
(b) `nearestBeat` returning a beat but `|instantDrift − drift| >
30 ms` (drift stale, instant fresh), or (c) `matchedOnsets`
demoted by the `consecutiveMisses` gate.

**Check 2 — Tight vs. loose match window sensitivity.** Add a
parametrized version of the Check 1 test that varies
`strictMatchWindow` across {30, 40, 50, 75} ms. For each, count
how many of the captured onsets register as strict-match vs.
search-only-match vs. miss. The output is a single small table
that says "at 30 ms, 40 % of onsets miss the strict gate; at 40 ms
it drops to 15 %". This tells you whether widening the strict
window (the cheapest possible fix) is sufficient or if the search
window itself is too tight.

This is a sensitivity sweep, not a fix. Do not commit a change to
the constant. The point is to know which knob has the most leverage.

**Check 3 — Decay-path inspection.** Read the no-onset decay logic
(should be near `step` / `update(playbackTime:)` — the path that
fires when no onset arrives within `2 × medianBeatPeriod`). On
Money, beats happen every ~487 ms (123.2 BPM); decay would only
fire if no onset arrives for ~975 ms. That's longer than expected
for a kick-driven track, but verify in the trace whether the
decay path is being entered. If decay-path entries correlate
with the plateau onset, that's a separate bug class than the
search-window-stale-drift hypothesis.
