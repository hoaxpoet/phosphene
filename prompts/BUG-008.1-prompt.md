────────────────────────────────────────
SYMPTOM
────────────────────────────────────────
DefaultBeatGridAnalyzer running over Love Rehab's 30-second Spotify
preview returns bpm=118.1, beats=59. The track is 125 BPM. That is a
5.5% systematic error. Five other tracks in the same session prepared
with plausible BPMs (Pyramid Song 70.0 vs true ~68, Money 123.2 vs
true ~120-125 in 7/4, etc.). Only Love Rehab is dramatically off
relative to a known-clean reference.

The error matters now because BUG-006.2's wiring fix means the
prepared grid is the one LiveBeatDriftTracker compares onsets against.
A 5.5% mistuned grid produces ~26 ms of drift per beat at a 480 ms
period; the strict-match window is ±30 ms and the search window is
±90 ms. Love Rehab's drift_ms walks linearly negative through both
windows and pegs at -90.490 by frame 2398 (31 s in) — the EMA freezes
at the search-window edge for the rest of the track and the drift
tracker is structurally unable to recover.

This bug has been latent since DSP.2 S5 (2026-05-04). It was masked
because engine.stemCache was never assigned (BUG-006 cause 1), so the
prepared grid was never installed at runtime — the live Beat This!
path on the 10-second tap window was filling in instead. We have no
record of whether that live path was more accurate on Love Rehab; it
may have been worse.

────────────────────────────────────────
DIAGNOSIS PATH
────────────────────────────────────────
Three orthogonal causes are plausible. Run all three checks; any one
that reproduces the problem narrows the fix surface dramatically.

Check 1 — Determinism. Re-prepare Love Rehab three times from fresh
launches. Compare BPMs. Outcomes:
  - All three return 118.1 (or within ±0.1) → deterministic.
    Eliminates non-determinism in resampling / mel filterbank
    quantization. The model is consistently producing this output for
    the input it's seeing.
  - BPMs vary by > 1 BPM across runs → non-determinism somewhere in
    the preprocessing chain. Most likely candidates are the
    AVAudioConverter resampler (default ratio precision can vary),
    mel filterbank floating-point ordering under vDSP, or
    MPSGraph dispatch-timing-sensitive ops.
  Save the three session.log files for evidence.

Check 2 — Sample-rate plumbing on the prepared-preview path.
Specifically: is the audio that lands in BeatThisPreprocessor.process
actually at 22050 Hz, or has the resampler delivered something
slightly off (which would cause the model to perceive the audio as
slowed)?
  - Add a one-time WIRING-style log at the top of
    BeatThisPreprocessor.process: input sampleCount, expected
    duration at 22050 Hz, log "WIRING: BeatThisPrep input
    sampleCount=N expected_22050_seconds=X.XX". Run on Love Rehab
    preview. The Spotify preview is ~30 s; if N / 22050 reads
    ~28.4 s, the resampler is producing audio 5.5% too slow.
  - Trace the upstream: PreviewDownloader decodes via AVAudioFile to
    [Float] — at what sample rate? Check whether the decoded buffer
    is at 44100 Hz or already at 22050 Hz. If the decoder is
    delivering 44100 Hz and BeatThisPreprocessor is the resampler
    site, verify AVAudioConverter is configured with
    sampleRateConverterAlgorithm = AVSampleRateConverterAlgorithm_Mastering
    (or a comparable high-quality setting) — the default fast
    converter has known accuracy issues.
  - This check has a blast radius beyond Love Rehab: if the
    resampler is mistuned, every prepared BPM is off by the same
    percentage. Cross-check the other tracks' offline BPMs against
    independent ground truth (a public BPM tool — see Check 3) and
    look for a consistent slope.

Check 3 — Independent BPM ground truth on the actual preview clip.
Fetch the exact 30-second Spotify preview Phosphene used for Love
Rehab (it lives in the same session directory if PreviewDownloader
caches the m4a — check the session capture's stems/ directory or
extract from raw_tap.wav if not). Run an independent BPM estimator
on it:
  - librosa.beat.beat_track (Python — the simplest reference)
  - aubio (alternate implementation)
  - Mixed In Key or RecordBox if you have them locally
  Outcomes:
  - All three independent tools say ~125 → the preview audio is
    correct; the failure is in our prepared path (resampler, model,
    or postprocessing).
  - Independent tools say ~118 → Spotify's preview clip is somehow
    different from the full track. Possibly a tempo-shifted or
    pitched preview (rare but documented for some tracks). In this
    case the bug isn't ours to fix; document and move on.
  - Spread (e.g. 122, 125, 119) → the preview itself is
    tempo-ambiguous. Less likely on a steady electronic track.

A small CLI tool can help here. The TempoDumpRunner CLI at
PhospheneEngine/Sources/TempoDumpRunner/ already accepts an audio
file and runs FFTProcessor + BeatDetector. It is not the same code
path as DefaultBeatGridAnalyzer (it uses the IOI-based DSP.1
estimator, not Beat This!) but it is independent third-party
verification by construction — Phosphene's own non-Beat-This! BPM
estimator. If TempoDumpRunner returns ~125 on the same preview
clip, that's strong evidence the preview is at 125 and the failure
is in Beat This! / its preprocessing.
