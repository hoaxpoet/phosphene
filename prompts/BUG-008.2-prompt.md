────────────────────────────────────────
DIAGNOSIS RECAP (read this carefully)
────────────────────────────────────────
BUG-008.1 proved the 5.5% BPM error on Love Rehab originates in the
upstream Beat This! model, not in any Phosphene code path. The
vendored PyTorch reference fixture (love_rehab_reference.json,
generated by the official Beat This! Python implementation on the
same audio) reports bpm_trimmed_mean=118.05; the Phosphene Swift
port returns 118.10 — within rounding. Three already-committed
regression tests pin every layer of the port: spectrogram match at
1e-3, layer-by-layer activations at 2e-3, resolver BPM at ±0.5.
Sample-rate plumbing, determinism, and resolver are all ruled out.

The "true 125 BPM" expectation came from the metadata tag in
CLAUDE.md / DSP.1 baseline, not from any upstream artifact. The
reference fixture's own description field uses "**~**125 BPM" — the
fixture author already knew the model produces 118.

**Open question: which BPM is actually correct?** The repo cannot
answer this. Three of four independent estimators on the same audio
(Beat This! PyTorch, Phosphene port, DSP.1 autocorrelation) agree on
~117–118; only the sub_bass IOI estimator returns ~125. The "125"
metadata tag almost certainly originates from kick-rate analysis
(Beatport / Spotify Audio Features / a DJ pool), which is the same
estimator class as DSP.1's sub_bass IOI — so the tag and DSP.1 are
not independent confirmations, they are one estimator and its tag.
Love Rehab may be a track where the kick fires at 125 BPM but the
*perceptual* beat (what Beat This! is trained to predict, since it
was trained on human tap annotations) integrates accent structure
across the mix and lands at 118. We do not know which is "right"
because "right" is not well-defined for music with this structure.

This is why the recommended fix is to surface the disagreement and
let the audio engineer make per-track calls, NOT to mechanically
prefer one estimator. Both estimators are doing the right thing for
their respective definitions of "the beat".

The 5.5% mistuned grid is what causes BUG-007's drift-window pegging
on Love Rehab specifically. BUG-007 (lock-hysteresis) is independent
and tracked separately.

────────────────────────────────────────
SCOPE — OPTION (1) FROM THE DIAGNOSIS WRITEUP
────────────────────────────────────────
Recommended scope: a verification gate that compares the two BPM
estimators already computed during preparation and emits a WARN log
when they disagree.

The two estimators are already in CachedTrackData:

  - TrackProfile.bpm — from analyzeMIR / BeatDetector / DSP.1
    trimmed-mean IOI on sub_bass. For Love Rehab this returns ~125.
  - CachedTrackData.beatGrid.bpm — from DefaultBeatGridAnalyzer /
    Beat This! / BeatGridResolver. For Love Rehab this returns 118.

Both run unconditionally during prepareTrack. Comparing them costs
nothing and surfaces the failure mode in session.log with no runtime
behaviour change.

Out of scope (do NOT do these in this increment):
  - Option (2) — feeding TrackProfile.bpm into BeatGrid when they
    disagree. Defer until BUG-007 is closed and we know what
    behaviour is actually broken vs. merely-disagreeing-cosmetically.
  - Option (3) — making LiveBeatDriftTracker re-estimate beat period
    from accumulated drift. Structural change to S7 invariants;
    tracked in QR.6 / a future increment.
  - Adding a UI surface for the disagreement (e.g. flashing the
    SpectralCartograph). The log line is sufficient for now.
  - Querying external BPM databases (Spotify audio-features, MIK,
    librosa subprocess). All add dependencies; none are needed.
  - Re-tuning Beat This! preprocessing or vendoring a different
    checkpoint. The diagnosis ruled out the port as the cause.

WHAT TO BUILD
────────────────────────────────────────
1. **Comparison + WARN log** in SessionPreparer's prepareTrack path.
   After both analyzeMIR and analyzeBeatGrid have run and produced
   CachedTrackData, compare TrackProfile.bpm against
   CachedTrackData.beatGrid.bpm. Both can be zero (analyzeMIR can
   fail silently; offline BeatGrid is .empty when nil analyzer or on
   model error). When BOTH are nonzero AND they disagree by more
   than the threshold, emit a single WARN-level os.Logger line.

   Threshold: 3% relative disagreement, computed as
   |a - b| / max(a, b) > 0.03. This is intentionally generous — the
   ±0.5 BPM resolver tolerance is ~0.4% at 125 BPM, so 3% leaves
   substantial headroom for legitimate small disagreements (e.g.
   123.2 vs 125 = 1.4% on Money would NOT fire). 5.5% on Love Rehab
   firmly does fire.

   Log format (mirrors the BUG-006.1 WIRING: convention but tagged
   for BUG-008's domain):

     WARN: BPM mismatch track='<title>' mir_bpm=<X.X> grid_bpm=<Y.Y>
       delta_pct=<Z.Z>% (BUG-008: upstream Beat This! disagrees with
       MIR estimator; prepared grid uses Beat This! value).

   The WARN goes to the same logger SessionPreparer already uses
   (Logging.session or the existing private logger in the file —
   match what's there). It also needs to land in session.log if
   SessionRecorder is active. Check whether the existing
   BUG-006.1 WIRING: lines reach session.log automatically through
   the existing logger plumbing; if not, add a sessionRecorder?.log()
   call alongside the os.Logger call so the warning is captured in
   the session capture for after-the-fact triage.

2. **Unit test coverage** in
   PhospheneEngine/Tests/PhospheneEngineTests/Session/. Use the
   existing SinePreviewDownloader / FixedURLResolver pattern from
   BeatGridIntegrationTests. Two tests minimum:

   - test_bpmAgreement_withinThreshold_doesNotWarn — construct a
     scenario where MIR and BeatGrid produce BPMs within 3% of each
     other (e.g. inject a stub BeatGridAnalyzing that returns a
     specific BPM near the MIR-derived BPM), assert no WARN.
   - test_bpmDisagreement_aboveThreshold_warns — construct a
     scenario where the gap exceeds 3% (e.g. stub BeatGridAnalyzing
     returns 118 while MIR analysis produces ~125 on a click-track
     fixture or stub), assert the WARN was emitted.

   Capturing os.Logger output in tests is awkward; the cleanest
   approach is to extract the comparison logic into a small pure
   function that returns an Optional<BPMMismatchWarning> struct, and
   unit-test that function directly. SessionPreparer calls the pure
   function, logs if the result is non-nil. Mirrors the existing
   pure-function-plus-thin-wrapper pattern used by
   FrameBudgetManager / MLDispatchScheduler.

3. **Live validation.** No manual capture required for this
   increment — the unit tests prove the comparison fires correctly,
   and the existing session capture 2026-05-06T20-11-46Z already
   contains the data that would trigger the log on Love Rehab.
   Reproducing the trigger live is good but not gating; if you do
   reproduce, attach the session.log excerpt to the closeout report.
