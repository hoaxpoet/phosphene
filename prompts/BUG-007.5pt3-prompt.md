Increment BUG-007.5 part 3 — BPM-aware lock-release gate.
Closes the HUMBLE-class half-time lock-flicker that BUG-007.5 parts 1+2
did not address.

Authoritative bug record: docs/QUALITY/KNOWN_ISSUES.md § BUG-007.5.
Predecessors: BUG-007.5 part 1 (time-based release, commit 832336ef),
BUG-007.5 part 2 (variance-adaptive tight gate, commit dc8cdb04).

This is a calibration fix, not a redesign. The lock-state semantic stays
exactly the same; only the release-window duration changes from a fixed
constant to a `max(floor, multiplier × medianBeatPeriod)` formula.

────────────────────────────────────────
DIAGNOSIS RECAP (read this before writing code)
────────────────────────────────────────
Manual session 2026-05-07T21-02-24Z showed BUG-007.5 part 2 closed lock
retention on most tracks (OMT 93 % LOCKED, MC 86 %, SLTS 94 %, Everlong
86 %) but HUMBLE held at 43 % LOCKED with a 5.4 s longest contiguous
run. The drift envelope was small (−109 to 0 ms, σ ≈ 15 ms), so the
adaptive tight gate from part 2 was already at floor. The problem was
the *release* gate, not the *match* gate.

Mechanism:
- HUMBLE's prepared grid is half-time (76 BPM, 790 ms beat period).
- The sub-bass onset detector occasionally returns nil during instru-
  mental breaks where the kick is masked by sustained bass or hi-hat.
- 3–4 consecutive nil-matches at 790 ms spacing accumulate ~3 s of
  non-tight time.
- The fixed `lockReleaseTimeSeconds = 2.5` gate fires before a tight
  match arrives → lock drops to .locking.
- Next kick re-acquires lock; cycle repeats every ~5 s.

Fast-tempo tracks (120+ BPM, period 500 ms) don't hit this — onset
density is high enough that consecutive misses rarely exceed 2 s.

The fix scales the gate with grid period: 4 × medianBeatPeriod. At
120 BPM (period 0.5 s), 4 × period = 2.0 s — clamped up to the 2.5 s
floor. At 76 BPM (period 0.79 s), 4 × period = 3.16 s — wider gate,
tolerates 4 sparse non-tight events. At 60 BPM (period 1.0 s), gate
reaches 4.0 s.

The 4× multiplier was chosen against the empirical onset-spacing data
in the 2026-05-07T21-02-24Z capture: HUMBLE's longest run of consec-
utive non-tight events was 4 across the locked window, so a gate that
admits 4 misses without dropping is the minimum that closes the
behavior. 5× would also work but adds risk of masking a genuinely-
broken lock; 4× is the smallest value that closes HUMBLE.

WHAT TO BUILD
────────────────────────────────────────
1. **Constant rename + new constant** in
   `PhospheneEngine/Sources/DSP/LiveBeatDriftTracker.swift`:

   ```swift
   /// Floor for the BPM-aware lock-release gate. At fast tempos
   /// (120+ BPM, 500 ms beat period) the gate stays at this floor —
   /// 2.5 s ≈ 5 beats. At slow tempos the gate scales up via
   /// `lockReleaseBeatMultiplier × medianPeriod`.
   private static let lockReleaseTimeSecondsFloor: Double = 2.5

   /// Beat-period multiplier for the BPM-aware lock-release gate.
   /// Lock drops only after 4 × medianBeatPeriod of non-tight events.
   /// At 120 BPM gives 2.0 s (clamped to 2.5 s floor); at 76 BPM
   /// half-time gives 3.16 s — enough for HUMBLE-class sparse
   /// onset streams that produce 3–4 consecutive non-tight events.
   private static let lockReleaseBeatMultiplier: Double = 4.0
   ```

   The old `lockReleaseTimeSeconds` constant is deleted.

2. **New private helper** alongside `computeLockState` /
   `computeLockStateLocked`:

   ```swift
   /// BPM-aware lock-release window. Returns the larger of:
   ///   - lockReleaseTimeSecondsFloor (2.5 s) for fast tracks,
   ///   - lockReleaseBeatMultiplier × medianPeriod (4 × period) for
   ///     slow tracks where 4 consecutive non-tight events are
   ///     not yet a "lost lock" signal.
   private func effectiveLockReleaseSecondsLocked() -> Double {
       let bpmAware = Self.lockReleaseBeatMultiplier * medianPeriod
       return max(Self.lockReleaseTimeSecondsFloor, bpmAware)
   }
   ```

   Caller invariant: holds the lock. `medianPeriod` is the existing
   private instance var, populated in `setGrid` from
   `newGrid.medianBeatPeriod` (with a 0.5 s default for empty grids).

3. **Wire the helper into `computeLockStateLocked(at:)`**: replace the
   `Self.lockReleaseTimeSeconds` reference with a call to
   `effectiveLockReleaseSecondsLocked()`. Update the doc comment to
   reflect the BPM-aware behavior.

4. **Two new tests** in
   `PhospheneEngine/Tests/PhospheneEngineTests/DSP/LiveBeatDriftTrackerTests.swift`:

NOTES FOR THE NEXT SESSION
────────────────────────────────────────
- The BUG-007 family is approaching closeout. After parts 1+2+3 of
  BUG-007.5 land, the remaining open items in the family are:
  BUG-007.7 (SLTS slow tempo drift, requires architectural rework —
  long-window adaptive grid; defer indefinitely or until Phosphene's
  beat-tracking architecture is revisited), and BUG-009 (halving
  threshold — still untouched, schedule when convenient).
- The 4× multiplier is intentionally conservative. If a future track
  shows lock-flicker on a tempo where 4 × period < 2.5 s but the
  flicker is still real (i.e. the floor is too tight), that's
  evidence to *raise the floor*, not raise the multiplier. The
  multiplier is for slow tracks; the floor is for fast tracks.
- Once parts 1+2+3 are validated, the natural next architectural
  question is: should the lock release gate be onset-count + time
  (whichever fires last) instead of time alone? An "AND of count
  and time" gate would be more permissive on extremely sparse tracks
  (HUMBLE breakdown sections with 1 onset every 2 s) but adds
  complexity. The current pure-time gate is simpler and adequate.
  Don't preemptively change.
- The `medianPeriod` instance var has a 0.5 s default for empty
  grids. That means the BPM-aware gate falls back to 2.0 s × 4 =
  2.0 s on no-grid (clamped to 2.5 s floor). Same behavior as
  pre-fix — an empty grid never enters .locked anyway because of
  the early `guard matchedOnsets >= lockThreshold` in
  `computeLockStateLocked`. No edge case to worry about.
- Do not consult BPM directly (`grid.bpm`) in the helper. Use
  `medianPeriod` because it's robust to slight beat-period jitter
  in the grid (the median of inter-beat intervals is more stable
  than the BPM scalar from the resolver). The two should be nearly
  equal but the median is the right primitive.
- If you find yourself wanting to make the multiplier configurable
  via a public property, stop. The floor is a tunable constant; the
  multiplier should not vary at runtime — it's an algorithmic
  invariant tied to the "4 misses is the threshold for genuine lock
  loss" decision.
