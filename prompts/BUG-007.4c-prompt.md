Increment BUG-007.4c — Auto-rotate refinement for kick-on-1+3 patterns.
Closes the residual `Shift+B` requirement on rock and hip-hop tracks
where the kick lands on slots 0 + 2 with similar density.

Authoritative bug record: docs/QUALITY/KNOWN_ISSUES.md § BUG-007.4
(extended with BUG-007.4c paragraph).
Predecessors: BUG-007.4a (`Shift+B` manual rotation, commit ee4b6a02),
BUG-007.4b (single-dominant auto-rotate, commit d1397055).

This is a calibration refinement, not a redesign. The auto-rotate gate
keeps its existing one-shot-per-track / manual-preempt semantics; only
the slot-selection logic gains a second path for the most common
real-music pattern.

────────────────────────────────────────
DIAGNOSIS RECAP (read this before writing code)
────────────────────────────────────────
Manual session 2026-05-07T21-35-22Z showed BUG-007.4b's auto-rotate
silently failed on the rock/hip-hop tracks the user wanted it for.
HUMBLE 56 % LOCKED, SLTS 92 %, Everlong 82 % — but the user reported
"still had to press Shift+B a bunch."

Cause: BUG-007.4b's `autoRotateDominanceRatio = 1.5` gate requires the
top slot to have ≥ 1.5× the runner-up's count. For rock and hip-hop
with kick on 1 *and* 3 of every 4/4 bar, slots 0 and 2 each accumulate
~50 % of the kicks. Slots 1 and 3 stay near zero (no kick on the
backbeat). Histogram: `[4, 0, 4, 0]`. Top : runner = 1.0. Gate
rejects. No rotation. User manually presses `Shift+B`.

The pattern is detectable from the histogram alone:
  - Top and runner-up have similar counts (within ~1.25× of each other)
  - The other slots sum to near-zero (the alternating-pattern signature)

Once detected, the slot to pick as "1" is one of {top, runner-up}.
Without snare-band data or kick-energy data, pure sub-bass onset count
can't distinguish slot 0 from slot 2. Tiebreaker: capture the slot of
the *first* tight onset on the track. Most listeners start playback
near a strong beat (intro kick, song's bar 1 beat 1), so the first
tight onset typically lands on the song's downbeat. Pick whichever of
{top, runner-up} matches that first-onset slot. Fall back to the
dominant slot if first-onset matches neither leader.

OMT-style four-on-the-floor (kick on every beat) produces histograms
like `[4, 4, 4, 4]` — top and runner-up are tied within ratio, but
the "other" slots are NOT near-zero (they hold the same density). The
alternating-pattern detector correctly rejects this case; manual
`Shift+B` remains the fallback for true 4-on-floor material.

SCOPE
────────────────────────────────────────
In scope:
- New tunable constants `autoRotateAlternatingTieRatio = 1.25` and
  `autoRotateAlternatingNoiseFraction = 0.20`. Add doc comments
  explaining what each value gates.
- New private state `firstTightOnsetRawSlot: Int?` — set on the very
  first tight onset of the current track, never updated again until
  `resetStateLocked` clears it.
- Reset hook for `firstTightOnsetRawSlot` in `resetStateLocked` (along
  with the other BUG-007.4b state).
- Wire the state capture into the existing tight-match branch of
  `update()`, immediately after the existing `slotOnsetCounts` push.
- Refactor `maybeAutoRotateBarPhaseLocked()` to delegate slot
  selection to a new private helper `chooseAutoRotateSlotLocked(...)`.
  The new helper implements both BUG-007.4b (single-dominant) and
  BUG-007.4c (alternating-pattern) selection in a single clear path.
- Three new tests covering: kick-on-1+3 with first onset on slot 2,
  kick-on-1+3 with first onset on slot 0, and the four-on-the-floor
  regression guard.

Out of scope:
- Multi-band onset signals (snare on 2/4, bass-stem energy). If the
  first-onset-slot tiebreaker ever picks wrong, the user can override
  with `Shift+B`. File a follow-up if data shows this is common.
- Changing the BUG-007.4b dominance ratio (1.5×) or the minimum
  dominant count (4). The new path supplements; it doesn't replace.
- Touching `firstTightOnsetRawSlot` outside the tight-match branch —
  must remain "first ever," not "most recent," to align with the
  tiebreaker's rationale.
- Onset-density signals from outside the sub-bass band. Same scope
  argument as above.
- Allowing auto-rotate to fire more than once per track. The existing
  `autoRotateAttempted` flag still gates everything; the new path
  shares it.

WHAT TO BUILD
────────────────────────────────────────
1. **Two new tunables** in
   `PhospheneEngine/Sources/DSP/LiveBeatDriftTracker.swift`, alongside
   `autoRotateMinDominantCount`:

   ```swift
   /// Tolerance for the kick-on-1+3 close-tie detection (BUG-007.4c).
   /// Top and runner-up are considered "tied" if their counts are
   /// within this ratio of each other (e.g. 4 vs 5 = 1.25× → tied).
   /// Most rock/hip-hop tracks have kick on 1+3 with slight emphasis
   /// on 1; allowing up to 1.25× covers normal variance while still
   /// detecting clear-winner cases (which take the original
   /// `autoRotateDominanceRatio` path).
   private static let autoRotateAlternatingTieRatio: Double = 1.25

   /// Floor (as fraction of top count) below which "other" slots are
   /// considered near-zero for kick-on-1+3 detection (BUG-007.4c).
   /// With 8 onsets on slots 0+2, slots 1+3 should sum to ≤ 20 % of
   /// slot 0's count to qualify as the alternating pattern.
   private static let autoRotateAlternatingNoiseFraction: Double = 0.20
   ```

2. **New private state** alongside `slotOnsetCounts`,
   `autoRotateAttempted`, `manualRotationPressed`:

   ```swift
   /// Raw slot (`beatsSinceDownbeat`) of the *first* tight onset in
   /// the current track. Used as the tiebreaker when the kick-on-1+3
   /// pattern is detected: two slots have similar high counts and the
   /// first tight onset usually lands on the song's downbeat.
   /// Reset to nil on `setGrid` / `reset`.
   private var firstTightOnsetRawSlot: Int?
   ```

3. **Reset hook** in `resetStateLocked()`:


4. **Capture point** in the tight-match branch of `update()`. After

5. **Refactor** `maybeAutoRotateBarPhaseLocked()` to delegate slot
   selection to a new `chooseAutoRotateSlotLocked(...)` helper. The
   new helper takes `dominantSlot`, `topCount`, `runnerUpSlot`,
   `runnerUp`, and `totalOnsets` and returns `Int?` (nil = leave the
   offset alone). It implements both paths:

   ```swift
   private func chooseAutoRotateSlotLocked(
       dominantSlot: Int, topCount: Int,
       runnerUpSlot: Int, runnerUp: Int,
       totalOnsets: Int
   ) -> Int? {
       // BUG-007.4b: clear single-dominant path.
       let dominanceFloor = max(Double(runnerUp), 1.0)
                          * Self.autoRotateDominanceRatio
       if Double(topCount) >= dominanceFloor {
           return dominantSlot
       }
       // BUG-007.4c: kick-on-1+3 alternating pattern.
       guard runnerUp >= Self.autoRotateMinDominantCount,
             runnerUpSlot >= 0 else {
           return nil
       }
       let tieRatioMet =
           Double(topCount) <= Double(runnerUp)
                             * Self.autoRotateAlternatingTieRatio
        && Double(runnerUp) <= Double(topCount)
                             * Self.autoRotateAlternatingTieRatio
       let othersCount = totalOnsets - topCount - runnerUp
       let noiseCeiling = max(
           2,
           Int(Self.autoRotateAlternatingNoiseFraction * Double(topCount))
       )
       let othersAreNoise = othersCount <= noiseCeiling
       guard tieRatioMet, othersAreNoise else { return nil }
       // Tiebreaker: first-tight-onset slot if it matches one of
       // {top, runner-up}; else default to dominant.
       if let first = firstTightOnsetRawSlot,
          first == dominantSlot || first == runnerUpSlot {
           return first
       }
       return dominantSlot
   }
   ```

   The caller (`maybeAutoRotateBarPhaseLocked`) tracks `runnerUpSlot`
   alongside `runnerUp` in its existing scan — initialise `runnerUpSlot`
   to `-1` and update it inside the existing `for idx in
   slotOnsetCounts.indices where idx != dominantSlot && ...` loop. Pass
   both into the helper. Apply the result the same way as before:

   ```swift
   _barPhaseOffset = ((bpb - chosen) % bpb + bpb) % bpb
   ```

   Update the auto-rotate log line to be path-agnostic (e.g.
   `"BUG-007.4 auto-rotate: counts=[…] chosen=… → offset=…"` rather
   than `"BUG-007.4b auto-rotate: …"`).

6. **Three new tests** in
   `PhospheneEngine/Tests/PhospheneEngineTests/DSP/LiveBeatDriftTrackerTests.swift`,

NOTES FOR THE NEXT SESSION
────────────────────────────────────────
- The BUG-007 family is essentially closed after this. Remaining open:
  - BUG-007.7 (SLTS slow tempo drift over long playback) — defer, requires
    architectural rework (long-window adaptive grid).
  - BUG-009 (halving threshold) — schedule when convenient.
- The `firstTightOnsetRawSlot` heuristic relies on listeners starting
  playback near a strong-beat moment. This generally holds for "play
  from start of track" but may be wrong if the user scrubs to a
  random offset (e.g. in the middle of a fill). The `Shift+B`
  override remains the fallback for those cases.
- Tunable `autoRotateAlternatingNoiseFraction = 0.20` is empirically
  derived from the 8-onset sample size. If the gate accumulates more
  onsets in practice (e.g. on slow tracks with longer time before the
  threshold trips), the noise fraction may need to scale down. Don't
  preemptively change.
- Do *not* widen `autoRotateDominanceRatio` from 1.5 → 1.2. The
  alternating-pattern path is the right way to handle the "ratio < 1.5
  but pattern is clear" case. Lowering the ratio admits noisy single-
  slot wins (e.g. one track where slot 0 has 5 hits and slot 1 has 4
  due to a stray hi-hat-as-bass detection — wrong rotation). Keep the
  paths semantically distinct.
- If a future session notices that `firstTightOnsetRawSlot` is stale
  on track changes (e.g. carries from track 1 into track 2), check
  `resetStateLocked` is being called from `setGrid`. The existing
  BUG-007.4b reset hook already nullifies it.
- The `chooseAutoRotateSlotLocked` helper signature accepts both
  `dominantSlot` + `runnerUpSlot` *and* their counts. This is by
  design — the caller already computed all four in its single-pass
  scan, and passing them avoids re-scanning the histogram in the
  helper. Don't refactor to "find them again in the helper" — that's
  duplicated work.
